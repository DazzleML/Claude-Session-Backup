# Human Test Checklist: `--` passthrough wrapper (#47)

**Target version:** v0.4.6
**Ships with commit:** `v0.4.6: feat: -- passthrough -- csb as a transparent wrapper (Closes #47)`
**Companion automated tests:** 12 in `tests/test_passthrough.py` (+ 3 view doubles updated); 947 total.

## What this is

`csb resume`/`csb view` now forward everything after a standalone `--` to the wrapped tool (claude / viewer), so the resolve-and-cd convenience extends to any flag the tool accepts. The automated tests mock the subprocess; only a human can confirm a **real claude launch** actually forks/cds correctly.

## High-Value Verification (~4 min)

### HV.1 -- The headline: resume + fork in one line
- [ ] `csb resume <a-session-name-or-prefix> -- --fork-session`
- [ ] **Expected:** prints `Resuming:` + `cd <start-folder>` + `claude --resume <uuid> --fork-session`, then claude actually launches **forked** (a new session branched off the original) **in the right folder**. This is the whole point -- confirm the fork really happened and the cwd is the session's start folder.

### HV.2 -- A second forwarded flag
- [ ] `csb resume <query> -- --model claude-opus-4-8` (or any claude flag you use)
- [ ] **Expected:** claude launches with that flag honored. (Confirms it's not special-cased to `--fork-session`.)

### HV.3 -- Isolation (forwarded flag is NOT eaten by csb)
- [ ] `csb resume <query> -- --db /tmp/should-not-be-used.db`
- [ ] **Expected:** csb uses your REAL index to resolve (the resolution succeeds/behaves normally); `--db /tmp/...` is handed to claude, not csb. (If csb tried to use that DB, resolution would fail/differ.)

### HV.4 -- Gate + empty cases
- [ ] `csb list -- foo` -> clear error, exit 2 ("passthrough is only supported by: resume, view"), nothing listed
- [ ] `csb resume <query> --` (nothing after) -> behaves exactly like a plain `csb resume <query>`

### HV.5 -- view passthrough (if you have a viewer)
- [ ] `csb view <query> -- <some-viewer-arg>` -> the viewer launches with the arg appended (or, no viewer installed, the transcript-path fallback still prints -- passthrough only matters when a viewer runs)

## What the automated tests DON'T cover
- A real `claude` process actually forking / honoring the forwarded flag / landing in the right cwd (HV.1-HV.3) -- the suite mocks `subprocess.run`.
- A real viewer binary receiving the args (HV.5).

## Cleanup
- None (no files written; HV.3's `/tmp/should-not-be-used.db` should NOT exist afterward -- if it does, isolation failed).
