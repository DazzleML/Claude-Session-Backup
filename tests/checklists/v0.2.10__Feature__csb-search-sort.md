# Human Test Checklist: csb search v0.2.10 (--sort + --full-info + readability)

**Target version:** v0.2.10
**Ships with commit:** v0.2.10: feat: csb search --sort + --full-info + readability polish
**Companion automated tests:** 38 new (9 in `test_search.py`, 12 in `test_cli.py`, 17 in `test_search_render.py`). Total 422/422 pass.

## What this is

Hand-runnable verification for the v0.2.10 `csb search` polish pass:
- `--sort {last-used,expiration,started,oldest,messages,size}` -- per-session iteration order, choices mirror `csb list --sort`
- `--full-info` / `-f` (level 1) -- adds `started: <date> (purge in Nd)` second header line per session
- `-ff` (level 2) -- escalates further to `start at: <folder> (Nx)` + top other folders + `N messages | vX.Y.Z` meta line, matching `csb list` shape
- Default header now shows human-readable relative date instead of raw ISO
- Session names render bold cyan (matches csb list/scan)
- Blank line between session blocks AND between thick metadata block + first hit (only at full-info >= 1)

## High-Value Verification (~3 minutes)

### HV.1 -- Default sort is unchanged

- [ ] `csb search "PUVM" --sessions-only --limit 10` AND `csb search "PUVM" --sessions-only --limit 10 --sort last-used` produce IDENTICAL output. (Default = last-used.)

### HV.2 -- `--sort oldest` reverses the order

- [ ] `csb search "PUVM" --sessions-only --sort oldest --limit 10` -- top session in the output is the OLDEST one (compare against `csb list --sort oldest`).

### HV.3 -- All six choices accepted

- [ ] Each of `last-used`, `expiration`, `started`, `oldest`, `messages`, `size` runs without argparse error:
  ```
  csb search "x" --sort last-used
  csb search "x" --sort expiration
  csb search "x" --sort started
  csb search "x" --sort oldest
  csb search "x" --sort messages
  csb search "x" --sort size
  ```

### HV.4 -- Invalid choice rejected at parse time

- [ ] `csb search "x" --sort bogus` exits with argparse error mentioning the choices.

### HV.5 -- Default header shows readable date (no raw ISO)

- [ ] `csb search "PUVM" --limit 1` -- the header line ends in something like `today (2026-05-17, at 10:27)` or `3 days ago (2026-05-14, at 18:00)`. There should be NO `last: 2026-05-...T...Z` raw ISO anywhere in the default human output.

### HV.6 -- `-f` / `--full-info` adds the second header line (level 1)

- [ ] `csb search "PUVM" -f --limit 1` -- second line under each session header reads `  started: <human date> (purge in Nd)` (purge half present if you have a cleanup period configured).
- [ ] `csb search "PUVM" --full-info --sessions-only --limit 1` -- same second line appears in sessions-only mode too, between the header and the `start at:` line.

### HV.6b -- `-ff` adds folder list + meta line (level 2)

- [ ] `csb search "PUVM" -ff --limit 1` -- under the `started:` line, more lines appear:
  - `  start at: <folder> (Nx)` -- start folder with its usage count
  - `  <other folder> (Nx)` -- up to 3 other folders the session visited
  - `  N messages | vX.Y.Z` -- meta line at the bottom
- [ ] A blank line separates this thick metadata block from the first hit (`L<n> [ROLE] ...`).
- [ ] `csb search "PUVM" -ff --sessions-only --limit 1` -- shows folder list + meta WITHOUT duplicating the `start at:` line (sessions-only already shows it with the resume hint).

### HV.7 -- Session names are bold cyan + visually separated

- [ ] `csb search "PUVM" --limit 5` -- session names render in **bold cyan** (same color as `csb list`'s session names).
- [ ] Two adjacent session blocks have a BLANK LINE between them; the second session's header doesn't run directly under the first session's last hit.

## Section 1: Sort semantics

### 1.1 --sort started shows newest-started sessions first

- [ ] Compare `csb search "x" --sessions-only --sort started` against `csb list --sort started` -- the top sessions should appear in the same order (when present in both).

### 1.2 --sort messages shows highest-message-count sessions first

- [ ] `csb search "x" --sessions-only --sort messages` -- the first session has the most messages of those containing matches.

### 1.3 --sort size shows largest JSONL first

- [ ] `csb search "x" --sessions-only --sort size` -- the first session has the largest `jsonl_size` of those containing matches.

### 1.4 --sort expiration shows soonest-to-purge first

- [ ] `csb search "x" --sessions-only --sort expiration` -- the first session is the one closest to its purge date (compare against `csb list --sort expiration`).

## Section 2: Interaction with other flags

### 2.1 --sort + --limit interaction

- [ ] `csb search "x" --sort oldest --limit 3` -- yields hits from at most 3 sessions, starting with the oldest-by-start-time one.

### 2.2 --sort + --sessions-only

- [ ] `csb search "x" --sessions-only --sort messages` -- summary mode respects the sort order (sessions in the summary appear in chosen order).

### 2.3 --sort + --files-only

- [ ] `csb search "x" --files-only --sort size` -- the unique source paths emerge in the chosen order (largest session's file first).

### 2.4 --sort + --json

- [ ] `csb search "x" --json --sort started` -- NDJSON output, but the first hit's session is the newest-started one.

### 2.5 --sort + --session-id filter

- [ ] `csb search "x" --session-id abc1,def2 --sort oldest` -- only those two sessions visited, ordered oldest-started first.

## Section 3: --full-info details

### 3.1 Purge half present when cleanup_days > 0

- [ ] Configure a cleanup period (or confirm one is set in `~/.claude/settings.json`'s `cleanupPeriodDays`). Run `csb search "x" --full-info --limit 1`. The second line should include `(purge in Nd)` where N is a positive integer.

### 3.2 Purge half absent when cleanup_days == 0

- [ ] If your settings have no cleanup period, the `started:` line should still appear but WITHOUT the trailing `(purge in ...)`.

### 3.3 Started date is human-readable

- [ ] The `started:` line shows `YYYY-MM-DD, at HH:MM` -- not raw ISO. Same format `csb list --sort started` uses.

### 3.4 --full-info in --sessions-only mode

- [ ] `csb search "x" --full-info --sessions-only` -- the `started:` line appears BETWEEN the header and the `start at:` line:
  ```
  <name>  <uuid>  (project)  -- N hits
    started: <date> (purge in Nd)
    start at: <folder>    [csb resume <uuid>]
  ```

## Section 4: Default human-readable date

### 4.1 No raw ISO in default output

- [ ] `csb search "x"` -- the header should NEVER include `T...Z` (raw ISO) in the default human mode.

### 4.2 --json still has raw ISO

- [ ] `csb search "x" --json | head -1` -- the `last_active_at` field IS the raw ISO. JSON output is the contract for machine consumers; readability changes don't touch it.

### 4.3 `last: ?` fallback for None last_active

- [ ] An old session with `last_active_at = NULL` should render as `last: ?` (not crash).

## Section 5: Visual separation + bold cyan name

### 5.1 Blank line between sessions

- [ ] `csb search "x" --limit 10` -- there is exactly one blank line between adjacent session blocks. No blank line BEFORE the very first session.

### 5.2 Bold cyan session name in default mode

- [ ] Session names render in bold cyan (visible in any ANSI-capable terminal). Same color csb list uses.

### 5.3 Bold cyan in --sessions-only mode too

- [ ] `csb search "x" --sessions-only` -- session names also bold cyan.

### 5.4 No color when --no-color

- [ ] `csb search "x" --no-color` -- no ANSI escapes, but the blank-line separator is still there.
