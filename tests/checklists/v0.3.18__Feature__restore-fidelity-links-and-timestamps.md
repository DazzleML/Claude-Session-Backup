# Human Test Checklist: Restore Fidelity -- All Symlinks Recreated + Original Timestamps

**Target version:** v0.3.18
**Ships with commit:** `v0.3.18: feat: restore fidelity -- recreate all symlinks + original timestamps` (Closes #39, Closes #40)
**Companion automated tests:** 146 tests in `tests/test_restore.py` (10 new for this release: 4 symlink-generalization, 6+1-rewritten timestamp-fidelity); 843 total in the suite.

## What this is

v0.3.18 makes `csb restore` byte+metadata-exact: every git symlink entry is recreated as a real link (#39, not just the logger's `transcript.jsonl`), and restored files get their original timestamps back (#40 -- mtime from the index / transcript events / git history, Windows creation time from the first event). This checklist verifies the parts mocks can't: real symlink privilege variance, real NTFS creation-time writes, file-manager date rendering, and the end-to-end "restored session is indistinguishable from never-deleted" experience.

## Prerequisites

1. csb v0.3.18 installed (`csb --version` shows `ALPHA 0.3.18`); `~/.claude` is a git repo with csb backups.
2. At least one **deleted** session in the index (`csb list --deleted -n 5`). Pick an EXPENDABLE one -- call its short id `<SID>` below.
3. On Windows: know whether this shell has symlink privilege (Developer Mode or admin). Both states are valid test paths.
4. Reset between sections: the restored files came FROM git, so removing them returns you to the deleted state. Prefer `dz safedel <path>` over raw deletion if available.

## High-Value Verification (~5 minutes)

### HV.1 -- Restored session carries OLD timestamps, not "just now"

Validates: #40's whole point -- recovery is metadata-exact.

- [ ] **Steps** (pick your shell; `<SID>` = your expendable deleted session):

  **PowerShell**:
  ```powershell
  csb restore <SID>
  Get-Item "$env:USERPROFILE\.claude\projects\*\<full-uuid>.jsonl" | Select-Object Name, CreationTime, LastWriteTime
  ```

  **cmd.exe**:
  ```cmd
  csb restore <SID>
  dir /T:W "%USERPROFILE%\.claude\projects\<slug>\<full-uuid>.jsonl"
  dir /T:C "%USERPROFILE%\.claude\projects\<slug>\<full-uuid>.jsonl"
  ```

  **POSIX**:
  ```bash
  csb restore <SID>
  stat ~/.claude/projects/*/<full-uuid>.jsonl
  ```
- [ ] **Expected:** output includes `Applied original timestamps to N file(s)`; LastWriteTime/mtime matches the session's last activity (compare `csb show <SID>`), NOT the current minute. On Windows, CreationTime matches the session's start.
- [ ] **Why this matters:** if this fails, every recovered session masquerades as brand-new in every file manager and age-based tool.

### HV.2 -- transcript.jsonl symlink exists and resolves after restore

- [ ] **Steps** (PowerShell / POSIX):
  ```powershell
  Get-Item "$env:USERPROFILE\.claude\sesslogs\*<SID>*\transcript.jsonl" | Select-Object LinkType, Target
  ```
  ```bash
  ls -l ~/.claude/sesslogs/*<SID>*/transcript.jsonl
  ```
- [ ] **Expected:** a SymbolicLink pointing at the restored `projects/<slug>/<uuid>.jsonl`; reading it shows transcript JSON. (No-privilege Windows: `Skipped N symlink(s)` reported instead, and NO regular file exists at that path.)
- [ ] **Why this matters:** a regular-file stub here permanently blocks the logger from relinking.

### HV.3 -- Dry-run reports everything and writes nothing

- [ ] **Steps** (cross-shell, csb only):
  ```
  csb restore <SID-of-another-deleted-session> --dry-run
  ```
- [ ] **Expected:** `Would restore N files`, `Would recreate N symlink(s) (... others verbatim from git)`, and `Would apply derived original timestamps to N file(s) ... Transcript mtime would be YYYY-MM-DD HH:MM:SS (from index).` No files appear on disk afterward.
- [ ] **Why this matters:** dry-run is the user's safety contract.

### HV.4 -- Restored session does NOT look "changed" to csb

Validates: #40's interaction with changed-detection and search (#36 trigger).

- [ ] **Steps** (cross-shell, csb only):
  ```
  csb backup --no-commit
  csb search "<a phrase you know is in the restored session>"
  ```
- [ ] **Expected:** backup does not re-flag the just-restored session as having un-backed-up changes; search finds the phrase WITHOUT a false-stale re-index of the restored session (the pre-v0.3.18 #36 trigger). Note: on a FRESH database, FTS5 must be built once first (`csb update build-fts5`) -- that one-time build is normal, not the false-stale bug this test guards against.
- [ ] **Why this matters:** pre-v0.3.18, the recovery-time mtime false-fired the FTS5 staleness check (#36) and made restores look like edits.

### HV.5 -- End-to-end: resume the restored session

- [ ] **Steps:** `csb resume <SID>` (or `claude --resume <full-uuid>` from the session's start folder)
- [ ] **Expected:** Claude Code loads the conversation; no "No conversation found".
- [ ] **Why this matters:** the whole tool's promise.

## Section 1: Symlink generalization (#39)

### 1.1 Verbatim recreate of a non-transcript symlink (synthetic)

- [ ] **Steps** (POSIX / privileged PowerShell): in a scratch clone of a test `~/.claude`, commit a symlink at `sesslogs/<dir-with-uuid>/extra-link.log` pointing anywhere, delete it on disk, run `csb restore <that-uuid>`.
- [ ] **Expected:** `Recreated N symlink(s) (transcript.jsonl -> the restored transcript; others verbatim).`; `readlink`/`Get-Item ... Target` shows the EXACT stored target text.
- [ ] **Check:** the link path is a real symlink, never a small regular file containing the target text.

### 1.2 No-privilege Windows fallback (run in a NON-elevated cmd without Developer Mode)

- [ ] **Steps:** same restore as HV.2 in an unprivileged shell.
- [ ] **Expected:** `Skipped N symlink(s) (could not recreate -- no symlink privilege?).`; restore exit code 0; transcript and all regular files intact; NO file at the link path.

### 1.3 Healing a blocking stub

- [ ] **Steps:** place a small regular file at a session's `sesslogs/<dir>/transcript.jsonl` path, then `csb restore <SID>` (privileged).
- [ ] **Expected:** stub replaced by a real symlink (LinkType reported); logger no longer blocked.

## Section 2: Timestamp fidelity (#40)

### 2.1 File-manager visual check (the user-story test)

- [ ] **Steps:** in Explorer (or Finder / a POSIX file manager), open the restored session's `projects/<slug>/` folder; sort by "Date modified", then "Date created" (Windows).
- [ ] **Expected:** the restored transcript sorts among its true contemporaries -- NOT at the top of the list.
- [ ] **Why mocks can't cover this:** rendering + sort behavior of a real file manager on real NTFS/ext4 metadata.

### 2.2 Git-fallback restore still derives times (no DB row)

- [ ] **Steps:** `csb update rebuild-index` on a scratch claude-dir (or delete the DB row), then `csb restore <full-uuid>` so the git-history fallback path runs.
- [ ] **Expected:** `(restored via git-history fallback ...)` AND `Applied original timestamps ...`; mtime equals the transcript's last event time (cross-check `csb show`).

### 2.3 POSIX behavior (Linux/macOS/BSD)

- [ ] **Steps:** repeat HV.1 on a POSIX box.
- [ ] **Expected:** mtime/atime restored via `os.utime`; NO creation-time claims (birth time is Windows-only; Unix ctime legitimately shows recovery time -- documented limitation); no pywin32 import errors.

### 2.4 Preserved files keep their live times

- [ ] **Steps:** restore a session that is fully present on disk (`csb restore <SID>` without `--force`).
- [ ] **Expected:** when ALL files are present: `Nothing to restore: all N expected files are already on disk.` When only SOME are present (mixed case): `Preserved N present file(s) ...`. In both cases: NO `Applied original timestamps` line for the preserved files; spot-check that their mtimes did not change.

## What the automated tests DON'T cover

- Real symlink privilege variance (Developer Mode on/off, elevation) -- tests skip or simulate
- Real NTFS creation-time writes via pywin32 on the user's actual volume
- File-manager date rendering and sort (Explorer/Finder)
- `claude --resume` against a real restored transcript
- FTS5 behavior on the user's real (large) search DB
- Performance feel on big restores (e.g. 196-file file-history sessions)
- POSIX platforms end-to-end (suite runs on Windows in CI here; #35 tracks the Linux walk)

## Reporting issues found

Note section + step, capture exact command and output, file an issue referencing this checklist and v0.3.18.

## Cleanup

- Synthetic scratch claude-dirs: remove entirely (prefer `dz safedel`).
- Real restored sessions you didn't want kept: the files all came from git -- remove the restored paths (prefer `dz safedel`) and the session returns to deleted state; `csb backup` will re-mark it deleted on next scan.
