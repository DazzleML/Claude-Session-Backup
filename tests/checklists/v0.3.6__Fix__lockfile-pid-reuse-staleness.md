# Human Test Checklist: lockfile PID-reuse staleness fix (v0.3.6)

**Target version:** v0.3.6
**Ships with commit:** v0.3.6: fix: reclaim stale backup lock on PID reuse
**Companion automated tests:** 18 in `tests/test_lockfile.py` (was 5; +13). Total 602/602.

## What this is

The backup lock (`~/.claude/.csb-backup.lock`) used to skip backups forever if its PID was reused after a crash/restart. v0.3.6 reclaims a stale lock by checking process *identity* (name + start-time) plus an age backstop, and makes the skip/reclaim visible. Three things to confirm by hand that mocks can't fully cover: real cross-platform process introspection, real lockfile lifecycle on disk, and the live messaging.

## High-Value Verification (~5 minutes)

### HV.1 -- Reclaim a lock held by a live-but-foreign PID (THE bug)

- [ ] **Steps (PowerShell):**
  ```powershell
  # Craft a lock claiming a real, alive, NON-csb process (this shell's PID)
  $pid_alive = $PID
  $lock = @{ pid = $pid_alive; acquired_at = (Get-Date).ToUniversalTime().ToString("yyyy-MM-ddTHH:mm:ssZ"); proc_name = "totally-not-csb.exe"; start_time = 1000.0; host = "repro" } | ConvertTo-Json -Compress
  $lock | Out-File -Encoding utf8 "$env:USERPROFILE\.claude\.csb-backup.lock"
  csb backup --no-commit
  ```
- [ ] **Expected:** prints `csb: reclaimed stale backup lock (PID <n>, PID reused by <name>) -- a prior backup was interrupted`, then the scan runs to completion. (The PID is alive but its real name isn't `totally-not-csb.exe`, so identity check #2 fires.)
- [ ] **Why it matters:** this is the exact failure that froze backups for two days. It must self-heal now.

### HV.2 -- Lock is released after a successful backup

- [ ] **Steps:** after HV.1, `Test-Path "$env:USERPROFILE\.claude\.csb-backup.lock"`
- [ ] **Expected:** `False` -- the lock is gone (released on context exit).

### HV.3 -- Dead-PID lock is reclaimed

- [ ] **Steps:**
  ```powershell
  '{"pid": 999999, "acquired_at": "2026-01-01T00:00:00Z", "proc_name": "python.exe", "start_time": 1.0}' | Out-File -Encoding utf8 "$env:USERPROFILE\.claude\.csb-backup.lock"
  csb backup --no-commit
  ```
- [ ] **Expected:** reclaims (reason `dead PID`) and runs. PID 999999 doesn't exist.

### HV.4 -- A genuinely concurrent backup is NOT reclaimed (informative skip)

- [ ] **Steps:** start a real backup in one terminal and, while it runs, launch a second:
  ```powershell
  # Terminal A (force it to take a moment on a large vault, or just race it):
  csb backup --no-commit
  # Terminal B, immediately:
  csb backup --no-commit
  ```
- [ ] **Expected:** Terminal B prints `Another csb backup is running (PID <n>, started <N>m ago). Skipping.` and exits 0 WITHOUT running. The live backup must NOT be reclaimed out from under itself.
- [ ] **Note:** if backups finish too fast to race, simulate: write a lock with `pid = $PID` (this shell, alive), a matching `proc_name` (run `(Get-Process -Id $PID).ProcessName + ".exe"`), and a fresh `acquired_at`; then `csb backup --no-commit` should skip.

### HV.5 -- Stale lock from a real prior restart is reclaimed on next backup

- [ ] **Steps:** if you ever see `Another csb backup is already running` when you know none is, run `csb backup` and confirm it now reclaims instead of skipping forever. (Pre-fix, this required manual lock deletion.)
- [ ] **Expected:** reclaim notice + scan runs.

### HV.6 -- `--quiet` suppresses the messaging (hook/cron contract)

- [ ] **Steps:**
  ```powershell
  '{"pid": 999999, "acquired_at": "2026-01-01T00:00:00Z"}' | Out-File -Encoding utf8 "$env:USERPROFILE\.claude\.csb-backup.lock"
  csb --quiet backup --no-commit
  ```
- [ ] **Expected:** no `reclaimed`/`Skipping` chatter on stderr; the backup still reclaims and runs silently.

## Section 1: lock file format

### 1.1 New lock is JSON

- [ ] While a backup runs (or right after writing one), inspect:
  ```powershell
  Get-Content "$env:USERPROFILE\.claude\.csb-backup.lock"
  ```
- [ ] **Expected:** a JSON object with `pid`, `acquired_at`, `proc_name`, `start_time`, `host` -- not a bare number.

### 1.2 Corrupted / old bare-PID lock is reclaimed

- [ ] **Steps:**
  ```powershell
  "12345" | Out-File -Encoding utf8 "$env:USERPROFILE\.claude\.csb-backup.lock"   # old bare-pid format
  csb backup --no-commit
  ```
- [ ] **Expected:** treated as corrupted/foreign and reclaimed (runs). No crash.

## Section 2: cross-platform introspection (run on each OS you have)

### 2.1 Windows -- name + start-time available

- [ ] `python -c "from claude_session_backup import lockfile; import os; print(lockfile._proc_name(os.getpid()), lockfile._proc_start_time(os.getpid()))"`
- [ ] **Expected:** a `python.exe`-ish name and a plausible epoch float (not None).

### 2.2 Linux -- `/proc` path

- [ ] Same command on Linux.
- [ ] **Expected:** a comm name (e.g., `python3`) and a float from `/proc/<pid>` ctime.

### 2.3 macOS / other -- graceful None

- [ ] Same command on macOS.
- [ ] **Expected:** `None None` is acceptable -- the age backstop still protects recovery (verify HV.3 still reclaims a dead PID).

## What the automated tests DON'T cover

- Real ctypes / `/proc` introspection on each OS (mocked in tests) -- Section 2
- Real two-process race for the same stale lock (unit test mocks the readback)
- Lock on a network-mounted `~/.claude` (mtime / readback semantics)
- A legit backup that genuinely runs > 30 min (would hit the age backstop) -- pathological; not exercised

## Cleanup

```powershell
# Remove any leftover repro lock:
Remove-Item "$env:USERPROFILE\.claude\.csb-backup.lock" -ErrorAction SilentlyContinue
# Run one clean backup to confirm a healthy steady state:
csb backup --no-commit
```
