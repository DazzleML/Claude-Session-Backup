# Human Test Checklist: csb build-fts5 (v0.3.0)

**Target version:** v0.3.0
**Ships with commit:** v0.3.0: feat: csb build-fts5 + FTS5 infrastructure (Phase 2 of #3)
**Companion automated tests:** 64 new (21 in `test_fts5_db.py`, 33 in `test_fts5_importer.py`, 10 in `test_fts5_index.py`). Total 487/487 pass.

## What this is

Hand-runnable verification for v0.3.0 Phase 2 infrastructure: per-project SQLite FTS5 content indices built from JSONL transcripts. `csb search` is intentionally unchanged in v0.3.0 -- this is the data-layer foundation; v0.3.1 will wire it into the search dispatcher.

## Prerequisites

1. csb v0.3.0 installed (`pip install -e ".[dev]"`)
2. `csb backup` has been run at least once (populates `sessions` + `session_sources`)
3. Local SQLite supports FTS5 (Python 3.10+ usually does; `csb build-fts5` will print a clear error otherwise)

## High-Value Verification (~5 minutes)

### HV.1 -- `csb build-fts5` runs cleanly from a fresh state

- [ ] **Steps:**
  ```
  csb build-fts5 --quiet
  ```
- [ ] **Expected:** exits 0; if not quiet, prints "FTS5 build: N indexed, 0 already fresh, N considered, M rows across P project(s)."
- [ ] **Check:** `~/.claude/csb-fts/` now exists and contains one `.db` file per project.

### HV.2 -- DBs contain expected role distribution

- [ ] **Steps (PowerShell):**
  ```powershell
  $db = (Get-ChildItem ~/.claude/csb-fts/*.db | Select-Object -First 1).FullName
  sqlite3 $db "SELECT role, role_subtype, COUNT(*) FROM messages GROUP BY role, role_subtype ORDER BY 3 DESC;"
  ```
- [ ] **Expected:** rows for USER, AI, and at least one AGENT subtype (e.g. `AGENT|explore|N`, `AGENT|dev-workflow-process|N`, etc. -- the variety depends on what skills/agents you used in that project).

### HV.3 -- File operations captured

- [ ] **Steps:**
  ```powershell
  sqlite3 $db "SELECT operation, COUNT(*) FROM file_operations GROUP BY operation;"
  ```
- [ ] **Expected:** counts for `read`, `edited`, `wrote`, `searched`, `notebook_edit` (whichever the project actually used). Bash deliberately not parsed.

### HV.4 -- Idempotency (re-run produces no work)

- [ ] **Steps:** run `csb build-fts5` twice in a row.
- [ ] **Expected:** second run says `N indexed: 0, K already fresh, ...` -- nothing got re-indexed.

### HV.5 -- `--force` triggers re-index

- [ ] **Steps:**
  ```
  csb build-fts5 --force
  ```
- [ ] **Expected:** every session counts in `sessions_indexed`; `sessions_skipped_fresh: 0`.

### HV.6 -- Project filter

- [ ] **Steps:** pick one project slug from `Get-ChildItem ~/.claude/projects/` and run:
  ```
  csb build-fts5 --project C--code-claude-projects-Claude-Session-Backup --force
  ```
- [ ] **Expected:** only that project's `.db` updated; other projects untouched (their mtime stays the same).

### HV.7 -- Session-id filter via shared resolver

- [ ] **Steps:**
  ```
  csb build-fts5 --session-id 7250ddce --force
  ```
- [ ] **Expected:** indexes exactly one session (the one whose UUID starts with `7250ddce`). On ambiguous prefix, prints candidate list + exits 2.

## Section 1: Schema integrity

### 1.1 messages_fts is an FTS5 virtual table

- [ ] `sqlite3 $db ".schema messages_fts"` includes `USING fts5` and `tokenize='porter unicode61'`.

### 1.2 Triggers keep FTS in sync

- [ ] Insert a test row manually:
  ```powershell
  sqlite3 $db "INSERT INTO messages (session_id, message_index, role, content) VALUES ('test', 999, 'USER', 'unique-test-phrase-xyz');"
  sqlite3 $db "SELECT count(*) FROM messages_fts WHERE messages_fts MATCH 'unique-test-phrase-xyz';"
  ```
- [ ] **Expected:** count = 1 (trigger fired).
- [ ] Clean up: `sqlite3 $db "DELETE FROM messages WHERE session_id = 'test';"` -- the FTS row should also disappear.

### 1.3 indexed_sessions has fresh rows after build

- [ ] `sqlite3 $db "SELECT session_id, last_jsonl_mtime, indexed_at FROM indexed_sessions LIMIT 3;"` returns rows with non-null timestamps.

### 1.4 file_operations is populated

- [ ] `sqlite3 $db "SELECT session_id, operation, file_path FROM file_operations LIMIT 5;"` returns rows showing actual file paths from your sessions.

## Section 2: AGENT subtype attribution

### 2.1 Skill-attributed AGENT rows

- [ ] `sqlite3 $db "SELECT DISTINCT role_subtype FROM messages WHERE role = 'AGENT' AND role_subtype IN ('commit','dev-workflow-process','fullpostmortem','prepcommit','obsidian');"` returns at least one row (assuming you've used any of those slash-command skills in this project).

### 2.2 Task-launched AGENT rows (from Agent tool sub-agents)

- [ ] `sqlite3 $db "SELECT DISTINCT role_subtype FROM messages WHERE role = 'AGENT' AND role_subtype IN ('explore','plan','oracle','senior-engineer','help','investigate');"` returns rows if you've spawned any Task-style agents.

### 2.3 AI rows (primary session responses)

- [ ] `sqlite3 $db "SELECT count(*) FROM messages WHERE role = 'AI';"` is large and positive (the primary assistant's own responses).

## Section 3: Mtime-driven re-index

### 3.1 Touching a JSONL triggers re-import

- [ ] **Steps:**
  ```powershell
  $jsonl = (Get-ChildItem ~/.claude/projects/C--code-claude-projects-Claude-Session-Backup/*.jsonl | Select-Object -First 1).FullName
  # Append an empty line so mtime advances
  Add-Content -Path $jsonl -Value ''
  csb build-fts5 --project C--code-claude-projects-Claude-Session-Backup
  ```
- [ ] **Expected:** the touched session shows up as re-indexed (sessions_indexed >= 1).

### 3.2 Subsequent run with unchanged mtime is idempotent

- [ ] Re-run the same command immediately -- sessions_indexed = 0, sessions_skipped_fresh = N.

## Section 4: Error handling

### 4.1 Missing JSONL file (referenced but absent)

- [ ] Edit `sessions.jsonl_path` for one row to a nonexistent path (sqlite3 direct UPDATE), then `csb build-fts5`. Build should silently skip that session, not crash.

### 4.2 FTS5 unavailable

- [ ] If you have access to a Python with non-FTS5 SQLite, `csb build-fts5` should exit 1 with a clear "FTS5 support" error message. (Not expected on standard Python 3.10+.)

### 4.3 Invalid --session-id prefix

- [ ] `csb build-fts5 --session-id xyz` (no match) → exits 1 with "No session found" message.
- [ ] `csb build-fts5 --session-id ab` (too short, < 4 chars) → exits 2 with "input ... is too short" message from the shared resolver.

### 5.1 Search behavior identical to v0.2.10

## What the automated tests DON'T cover

- Real `~/.claude` vault performance under 100+ sessions
- Disk-space behavior of per-project DBs over time
- Concurrent `csb build-fts5` invocations (file locking)
- FTS5 tokenizer edge cases for non-ASCII content in real transcripts
- Interaction with the existing `csb backup` hook (deliberately none in v0.3.0)
- Visual `sqlite3 <db>` inspection -- humans verify this hand

## Reset between sections

Tests are read-mostly. To reset between runs:
```powershell
Remove-Item ~/.claude/csb-fts/*.db   # purges all per-project FTS5 indices
```
Next `csb build-fts5` will rebuild from scratch.
