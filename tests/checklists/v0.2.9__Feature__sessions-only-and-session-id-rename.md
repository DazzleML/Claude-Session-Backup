## High-Value Verification (~5 minutes)

### HV.1 -- `--sessions-only` produces a session summary, not excerpts

Validates: new output mode renders correctly and groups hits by session.

- [ ] **Steps:**
  ```
  csb search "PUVM" --sessions-only --limit 50
  ```
- [ ] **Expected:** one summary block per session (NOT excerpts). Each block has a header line ending in `-- N hits` (or `-- 1 hit` singular), then a second line `  start at: <folder>    [csb resume <full-uuid>]`. A trailing `Next: csb search ...` hint appears at the bottom.
- [ ] **Why this matters:** the whole point of the mode -- "which sessions contain X" without scrolling excerpts.

### HV.2 -- `--session-id` accepts comma-separated UUID prefixes

Validates: rename + multi-prefix OR-match work end-to-end.

- [ ] **Steps:** Pick two real UUIDs from `csb list -n 3` output. Note their first 4 chars (P1, P2). Then:
  ```
  csb search "anything" --sessions-only --session-id P1,P2
  ```
- [ ] **Expected:** results filtered to ONLY those two sessions (if either has matches).
- [ ] **Why this matters:** the whole multi-session drill-in workflow depends on this.

### HV.3 -- `--session` is rejected (rename complete)

Validates: the breaking-change rename is enforced.

- [ ] **Steps:**
  ```
  csb search "anything" --session abc1
  ```
- [ ] **Expected:** argparse error along the lines of `unrecognized arguments: --session abc1`. Exit code != 0.
- [ ] **Why this matters:** confirms the old flag is gone, no silent acceptance.

### HV.4 -- Output-mode mutex group rejects combinations

Validates: argparse refuses two output modes at once.

- [ ] **Steps:** Each of these should fail at parse time:
  ```
  csb search "x" --json --files-only
  csb search "x" --json --sessions-only
  csb search "x" --files-only --sessions-only
  ```
- [ ] **Expected:** each command exits non-zero with `argument ...: not allowed with argument ...`.
- [ ] **Why this matters:** preventing user confusion about which mode they got.

### HV.5 -- `--sessions-only --shortid` uses compact UUID end-to-end

Validates: the shortid plumbing extends to the new mode + resume hint.

- [ ] **Steps:**
  ```
  csb search "PUVM" --sessions-only --shortid --limit 5
  ```
- [ ] **Expected:** the header UUID column shows `<head>-...-<tail>` form, AND the inline `[csb resume <uuid>]` hint also uses the compact form (consistent paste-back).
- [ ] **Why this matters:** the compact form must thread through every renderer field uniformly.

### HV.6 -- Default `--sessions-only` raises the effective limit

Validates: a single noisy session doesn't crowd others out of the summary.

- [ ] **Steps:**
  ```
  csb search "the" --sessions-only
  ```
  Pick a common word likely to appear many times in many sessions.
- [ ] **Expected:** dozens of sessions in the summary (NOT just one or two). The default `--limit 20` is automatically bumped in sessions-only mode.
- [ ] **Why this matters:** the semantic is "which sessions contain this" -- artificial truncation defeats the purpose.

## Section 1: --session-id (rename + multi-prefix)

### 1.1 Single prefix still works (backward-compat with single-value usage)

- [ ] `csb search "anything" --session-id <prefix>` constrains to one session, same as old `--session`.

### 1.2 Comma-separated multi-prefix

- [ ] `csb search "x" --session-id P1,P2,P3` OR-matches three sessions.

### 1.3 Trailing/leading commas tolerated

- [ ] `csb search "x" --session-id ,abc1,,def2,` works the same as `--session-id abc1,def2`.

### 1.4 Whitespace tolerated

- [ ] `csb search "x" --session-id " abc1 , def2 "` strips whitespace before matching.

### 1.5 4-char minimum (resolver-level)

- [ ] `csb search "x" --session-id abc` matches NO sessions (3 chars is below the minimum; the SQL LIKE 'abc%' may still match, but the design intent is at-least-4 -- check that the actual behavior is documented and consistent).
  - **Note**: if 3-char prefixes accidentally work, that's a leak from the resolver's minimum check. File as a follow-up if observed.

## Section 2: --sessions-only formatting details

### 2.1 Per-session block shape (default, full UUID)

- [ ] Each block has exactly 2 lines (header + start-at) plus a blank-line separator.
- [ ] Header line: `<name>  <full-uuid>  (<project>)  -- N hits` (or "1 hit" singular).
- [ ] Second line: `  start at: <folder>    [csb resume <full-uuid>]`.

### 2.2 With --shortid

- [ ] Header UUID and `[csb resume ...]` hint both use compact `<head>-...-<tail>` form.
- [ ] Both are the SAME compact form (no mixing).

### 2.3 Singular vs plural grammar

- [ ] Sessions with exactly 1 hit say `-- 1 hit` (not "1 hits").
- [ ] Sessions with 2+ hits say `-- N hits`.

### 2.4 Unknown start_folder fallback

- [ ] If a session's `start_folder` is NULL (rare but possible for old sessions), the start-at line shows `(unknown)` instead of crashing.

### 2.5 Next: hint at the end

- [ ] After all sessions, a blank line then `Next: csb search "<original query>" --session-id <first-uuid> -A 3 -B 1`.
- [ ] The `<original query>` is the exact pattern the user typed (quoted).
- [ ] The `<first-uuid>` matches the format of the headers above (full or compact per --shortid).

### 2.6 Iteration order preserved

- [ ] Sessions appear in `last_active_at DESC` order (newest first), matching the order in `csb list`.

## Section 3: Mutex enforcement

### 3.1 No flag (default mode)

- [ ] `csb search "x"` renders the standard grouped-excerpt human output.

### 3.2 Single mode flags work

- [ ] `csb search "x" --json` → NDJSON.
- [ ] `csb search "x" --files-only` → unique paths only.
- [ ] `csb search "x" --sessions-only` → per-session summary.

### 3.3 Pairwise combinations rejected

- [ ] `--json --files-only` → SystemExit, argparse error.
- [ ] `--json --sessions-only` → SystemExit, argparse error.
- [ ] `--files-only --sessions-only` → SystemExit, argparse error.

### 3.4 Triple combination rejected

- [ ] `--json --files-only --sessions-only` → SystemExit, argparse error (first conflict reported).

## Section 4: Interaction with other flags

### 4.1 --sessions-only + --regex

- [ ] `csb search -E "oauth.*flow" --sessions-only` works -- regex matching, summary output.

### 4.2 --sessions-only + --source jsonl

- [ ] `csb search "x" --sessions-only --source jsonl` forces JSONL parser, summary output.

### 4.3 --sessions-only + --all / --deleted

- [ ] `csb search "x" --sessions-only --all` includes deleted sessions in the summary.
- [ ] `csb search "x" --sessions-only --deleted` shows only deleted sessions.

### 4.4 --sessions-only + --no-color

- [ ] `csb search "x" --sessions-only --no-color` works -- no ANSI escapes in output.

### 4.5 Empty results

- [ ] `csb search "definitely-no-such-string-12345" --sessions-only` prints the standard "No content matches" message + hint, exits 0.
