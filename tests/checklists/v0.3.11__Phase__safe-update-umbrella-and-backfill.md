# Human Test Checklist: v0.3.11 -- `csb update` umbrella + safe rebuild + backfill

**Target version:** 0.3.11
**Ships with commit:** TBD (pre-commit checklist)
**Companion automated tests:** 770 tests passing across `tests/` (758 prior + 12 new for v0.3.11: snapshot/restore safety, .bak crash recovery, --include-fts5 stub seam, `git_list_deleted_jsonls` path-prefix handling, cache CRUD helpers, schema v3->v4->v5 migrations, val: format, PURGED wording, width-aware layout, cmd_backfill_deleted end-to-end).

## What this is

A hand-runnable validation pass for v0.3.11's two big stories:

1. **Safe `csb update rebuild-index`** -- the destructive pre-v0.3.11 implementation that silently lost deleted-session metadata is replaced with a snapshot-then-merge dance + `.bak` crash recovery + `backup_lock` concurrency safety.
2. **`csb update backfill-deleted`** -- discover deleted sessions from git history and synthesize / auto-repair their metadata.

Plus user-visible polish: the `csb update` umbrella (replacing top-level `csb rebuild-index` / `csb build-fts5`), the `val: YY-M-D` display field signaling "we cross-checked this row," PURGED vs OVERDUE wording for past-cleanup sessions, and width-aware two-line layout for deleted-session footers.

External testers: this checklist is self-contained -- you don't need to read the design docs to run it.
Project maintainers: see [Planning lineage](#planning-lineage) for design context.

## Prerequisites

1. **Windows 10/11 or Linux/macOS**. Most user-impact tests are Windows-relevant (the original data-loss bug surfaced there); regression sections also run on POSIX.
2. **csb v0.3.11 installed in editable mode**:

   **cmd.exe** / **PowerShell** / **POSIX** (same command):
   ```
   pip install -e .
   ```
3. **A scratch `~/.claude/` lookalike** for destructive tests. Two options:
   - **Option A (safer, recommended)**: copy your real DB to a scratch path and use `--db <path>` on every csb command. Sections that need a fake `~/.claude/` git repo will use `tests/one-offs/rebuild_reality_check.py` (sandboxed in `TemporaryDirectory`).
   - **Option B**: throwaway `~/.claude/` clone in `%TEMP%` / `/tmp`. Slower setup but exercises the real hook path.
4. **GPG agent running** (Kleopatra on Windows) -- only matters if you intend to commit. Skip for read-only checklist passes.

Always use `--db <path>` to point csb at a scratch DB rather than your real `~/.claude/session-backup.db`. The Section 0 setup commands establish this isolation.

### HV.1 -- `csb update` umbrella exists; old top-level verbs are gone

  ```
  csb --help
  ```

  - Each `csb update <target> --help` works and shows the right description
  - `csb rebuild-index` and `csb build-fts5` fail with "invalid choice"
- [ ] **Why this matters:** If the dispatcher is wrong, every subsequent test fails. This is the load-bearing CLI surface change.

### HV.2 -- Safe rebuild preserves deleted-session metadata

Validates: the data-loss bug is fixed.

- [ ] **Steps** (cross-shell):
  ```
  csb --db %CSB_SCRATCH%\test.db list --deleted only -n 3
  csb --db %CSB_SCRATCH%\test.db update rebuild-index
  csb --db %CSB_SCRATCH%\test.db list --deleted only -n 3
  ```
  *(POSIX/PowerShell: substitute the right env var form; commands are equivalent.)*
- [ ] **Expected:**
  - Pre-rebuild list shows N deleted sessions with their folder paths.
  - `csb update rebuild-index` prints `Moved old index aside: <db>.bak`, then `Preserved N deleted-session record(s) across rebuild`, then `Index rebuilt successfully.`
  - Post-rebuild list shows the SAME N deleted sessions with the SAME folder paths and the same `message_count`.
  - No `.bak` file remains afterward (cleaned up on success).
- [ ] **Why this matters:** Pre-v0.3.11 this rebuild silently dropped every deleted-session row. The fix is the headline of this release.

### HV.3 -- `csb update backfill-deleted` runs cleanly and stamps val:

Validates: backfill works end-to-end and the new `val:` display field appears.

- [ ] **Steps** (cross-shell):
  ```

  ```
- [ ] **Expected:**
  - backfill prints `git log: N deleted JSONL path(s); M new to cache` then `backfill-deleted: synthesized X session(s), repaired Y sparse row(s) (Z intact in live DB, W unreadable from git)`.

  - For previously-sparse sessions (single-folder rows), `folder_usage` may have grown -- check by re-running `csb list --deleted only --all-folders` for one of them.
- [ ] **Why this matters:** backfill is the second-half of the recovery story. `val:` is the new "we've checked this" signal.

### HV.4 -- PURGED wording replaces OVERDUE for already-deleted sessions

Validates: the OVERDUE-by-Nd display polish ships and works on real data.

- [ ] **Steps:**

- [ ] **Why this matters:** "OVERDUE" reads as "should have happened but didn't" -- inaccurate when the cull already happened.

- [ ] **Steps:** Pick a deleted session UUID with multiple `folder_usage` rows. Pre-rebuild: query the count. Run rebuild. Post-rebuild: query the count.

### 2.3 `backup_lock` blocks concurrent backup

- [ ] **Steps:**

- [ ] **Expected:** Shell B prints `Another csb backup or update is in progress. Rebuild skipped.` and exits 0.
- [ ] **Note:** Skip this if you can't easily simulate the concurrency window. The automated test covers the same lock contract via mocks.

## Section 3: `csb update backfill-deleted`

Reset scratch DB before this section (same commands as Section 2).

### 3.1 First-run output structure

- [ ] **Steps:**
  ```
  csb --db %CSB_SCRATCH%\test.db update backfill-deleted
  ```
- [ ] **Expected:** Two lines:
  - `git log: N deleted JSONL path(s); M new to cache`
  - `backfill-deleted: synthesized X session(s), repaired Y sparse row(s) (Z intact in live DB, W unreadable from git)`
- [ ] **Note on `M new to cache`:** if the scratch DB was copied from a `~/.claude/session-backup.db` whose `git_deleted_jsonls` cache table has already been populated by a prior v0.3.11 backfill run, `M` will be `0` (cache already has the rows). On a truly fresh DB (or post-`rebuild-index` if no chained backfill ran), `M` should equal `N`. Either is correct.

### 3.2 `--dry-run` writes nothing

- [ ] **Steps:**
  ```
  copy %CSB_SCRATCH%\test.db.snapshot %CSB_SCRATCH%\test.db /Y
  csb --db %CSB_SCRATCH%\test.db update backfill-deleted --dry-run
  python -c "import sqlite3; c=sqlite3.connect(r'%CSB_SCRATCH%\test.db'); print('cache rows after dry-run:', c.execute('SELECT COUNT(*) FROM git_deleted_jsonls').fetchone()[0])"
  ```
- [ ] **Expected:** Cache has 0 rows after dry-run. Dry-run output should print the would-be repair/synthesize summary without actually writing.

### 3.3 Auto-repair runs on sparse rows whose git blob has more folders

- [ ] **Steps:** Find a deleted session with `folder_usage` count <= 1 AND whose git blob (per the conditions in `docs/maintenance.md`) has multiple cwds.
  ```
  python -c "import sqlite3; c=sqlite3.connect(r'%CSB_SCRATCH%\test.db'); rows=c.execute('SELECT s.session_id, COUNT(f.folder_path) FROM sessions s LEFT JOIN folder_usage f ON f.session_id=s.session_id WHERE s.deleted_at IS NOT NULL GROUP BY s.session_id ORDER BY 2 LIMIT 5').fetchall(); [print(r) for r in rows]"
  csb --db %CSB_SCRATCH%\test.db update backfill-deleted
  ```
- [ ] **Expected:** `repaired N` count > 0 if any sparse-with-richer-blob rows existed. After backfill, re-query those UUIDs -- their `folder_usage` count should be larger than before.

### 3.4 Idempotent: second run is a no-op

- [ ] **Steps:**
  ```
  csb --db %CSB_SCRATCH%\test.db update backfill-deleted
  csb --db %CSB_SCRATCH%\test.db update backfill-deleted
  ```
- [ ] **Expected:** Second run shows `synthesized 0, repaired 0` and the same intact/unreadable counts as run 1. No duplicate sessions rows.

### 3.5 CLI dispatcher routes correctly end-to-end

- [ ] **Steps:**
  ```
  csb --db %CSB_SCRATCH%\test.db update backfill-deleted --quiet
  ```
- [ ] **Expected:** Exit 0, no output (quiet mode).

## What the automated tests DON'T cover

- **Shell rendering**: Windows codepage 437/1252, ANSI color codes, terminal width detection in real (non-pipe) terminals
- **Real git operations against a real `~/.claude/` repo**: tests use sandboxed temp repos; the real repo's history is much larger and may have edge cases (renames, branches, etc.)
- **Performance on large DBs**: tests have 10s of sessions; real DBs have 100s-1000s
- **Interactive prompts**: backfill doesn't prompt currently, but `csb scan --restore` does (covered by its own checklist)
- **Cross-platform paths**: tests primarily use Windows-style paths; POSIX regression is one test
- **Hook integration**: tests don't exercise `csb update` via the SessionStart/PreCompact hook path
- **`.bak` crash recovery in the wild**: hard to simulate a real crash mid-rebuild; the automated test uses `monkeypatch` to raise

## Reporting issues found

1. Note the section and step number (e.g., "Section 2.3 failed").
2. Capture the exact command and output.
3. Save the `test.db` and any `.bak` files to a stable path.
4. File a GitHub issue referencing this checklist and the affected version (v0.3.11).
5. Link back to the commit hash when known.

## Cleanup

Remove the scratch directory at the end:

**cmd.exe**:
```cmd
rmdir /s /q %CSB_SCRATCH%
set CSB_SCRATCH=
```

**PowerShell**:
```powershell
Remove-Item -Recurse -Force $env:CSB_SCRATCH
Remove-Item Env:\CSB_SCRATCH
```

**POSIX**:
```bash
rm -rf "$CSB_SCRATCH"
unset CSB_SCRATCH
```

Your real `~/.claude/session-backup.db` was never modified -- all destructive operations targeted the scratch copy.
