# Human Test Checklist: Short-UUID Resolver and Display Sugar

**Target version:** v0.2.7
**Ships with commit:** `v0.2.7: feat: short-UUID resolver + display sugar across csb` (pre-commit at time of writing)
**Companion automated tests:** 23 new tests in `tests/test_ids.py` (resolver tier-by-tier, display helper, candidate formatting), plus updated `tests/test_commands.py::test_resume_session_not_found_returns_1_no_subprocess` to verify the resolver error path. Full suite: 350/350 pass.

## What this is

A hand-runnable checklist to verify that the short-UUID sugar works end-to-end on a real Claude Code installation. Automated tests cover the resolver's three-tier matching logic, the display helper's truncation, and the ambiguous-error formatter — but they all run against synthetic in-memory rows. This checklist exercises the real workflow ("I found a session, now load into it"), the real cross-shell behavior, the readability of the new compact UUID display, and the exit-code contract that scripts may eventually rely on.

## Prerequisites

1. **csb v0.2.7 installed** (editable install or PyPI when published):
   - `pip install -e .` from the repo root, or
   - `pip install claude-session-backup==0.2.7`
2. **At least 5 indexed Claude Code sessions** in `~/.claude/projects/`. Run `csb backup --no-commit` once if you've never indexed the DB. (If you have fewer than 5, some collision tests are trivially uninteresting — but every other test still works.)
3. **A working `claude` binary on PATH** (only needed for the `csb resume` integration test in Section 5).
4. **One of: cmd.exe, PowerShell, or bash/zsh.** This checklist provides commands in all three forms.

**State isolation**: this checklist is **read-only against your real `~/.claude/session-backup.db`**. None of the steps below modify the DB or the session JSONLs. The only mutating step is `csb resume` in Section 5, which launches `claude --resume` — that's `claude`'s I/O, not csb's, and it doesn't change the JSONL.

**Reset between sections**: no reset needed — every command is idempotent and read-only against csb's state.

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these six tests. They cover the highest-impact user-visible behaviors this version introduces. If any of them fail, the work is broken.

### HV.1 — `csb show` accepts an 8-char prefix

Validates: the most common short-UUID use case — `csb show <first-8-chars>` resolves to a single session.

- [ ] **Steps:**

  1. Pick any session and copy its 8-char prefix from `csb list -n 1`:

     **cmd.exe**:
     ```cmd
     csb list -n 1
     ```

     **PowerShell**:
     ```powershell
     csb list -n 1
     ```

     **POSIX**:
     ```bash
     csb list -n 1
     ```

     The `id:` line shows the new compact form: `id: 7250ddce-...-3d73adaf`. Copy the 8-char head (`7250ddce` in this example).

  2. Run `csb show <8-char-prefix>`.

- [ ] **Expected:** Detailed session info prints with the full UUID on the `ID:` line. No "No session found" error.
- [ ] **Why this matters:** If this fails, the resolver isn't being called or its prefix tier is broken — the headline feature doesn't work.

### HV.2 — `csb show` rejects too-short input with a clear message

Validates: the 4-char minimum-length guard prints a helpful error rather than degrading to "guess from many candidates."

- [ ] **Steps:**
  1. Run `csb show abc` (3 chars).
- [ ] **Expected:** stderr message: `Error: Session ID input 'abc' is too short (need at least 4 hex chars)`. Exit code 2.
- [ ] **Why this matters:** Without this guard, every 1-3 char input would silently match many sessions and the resolver's collision UX would dominate the user experience. The message must be helpful, not generic.

### HV.3 — `csb show` accepts a UUID suffix

Validates: the third resolver tier (suffix match) works when a prefix wouldn't be useful — e.g., copying the trailing portion of a UUID.

- [ ] **Steps:**
  1. From the same `csb list -n 1` output above, copy the 8-char tail shown (e.g. `3d73adaf`).
  2. Run `csb show <8-char-tail>`.
- [ ] **Expected:** Same session as HV.1 displays. No "No session found".
- [ ] **Why this matters:** Suffix matching is the disambiguator for cases where the prefix tier collides. If suffix matching is broken, users get stuck having to type the full UUID.

### HV.4 — `csb list` displays the FULL UUID by default

Validates: the displayed UUID can be pasted directly into `claude --resume <uuid>` (the native claude binary has no resolver, so it needs the full form).

- [ ] **Steps:**
  1. Run `csb list -n 3`.
- [ ] **Expected:** Every session row shows `id: <full-36-char-UUID>` (e.g. `id: 7250ddce-6d73-4e30-9ed3-c6793d73adaf`), NOT the compact form.
- [ ] **Why this matters:** If the default display were the compact form, users couldn't copy what they see into native tools like `claude --resume`. csb is supposed to amplify the existing workflow, not break it.

### HV.5 — `csb list --shortid` opts in to the compact UUID display

Validates: the `--shortid` flag (also `-sid`) switches the same `id:` field to the compact `<head>-...-<tail>` form.

- [ ] **Steps:**
  1. Run `csb list -n 3 --shortid`.
  2. (Equivalent: `csb list -n 3 -sid`.)
- [ ] **Expected:** Every session row shows `id: <8-head>-...-<8-tail>` (e.g. `id: 7250ddce-...-3d73adaf`).
- [ ] **Also check:** `csb search <q> --shortid` similarly switches its session-header display to compact. (`csb show`'s detail view always displays the full UUID -- by design, no flag.)
- [ ] **Why this matters:** This is the opt-in path for the compact form, and it's the same `--shortid` flag across `csb list`, `csb scan`, and `csb search` -- consistent UX.

### HV.5b — `csb show` accepts the compact display form (paste-back round-trip)

Validates: a user can copy the compact ID shown by `csb list --shortid` or `csb search --shortid` and paste it back into any csb command. Without this, the compact form would feel like a dead-end display rather than a usable handle.

- [ ] **Steps:**
  1. Run `csb list -n 1 --shortid`.
  2. Copy the entire compact ID from the `id:` line, including the `...`. Example: `7250ddce-...-3d73adaf`.
  3. Run `csb show 7250ddce-...-3d73adaf` (with the captured value).
- [ ] **Expected:** The session detail prints normally with the full UUID on the `ID:` line.
- [ ] **Why this matters:** Earlier in development the resolver rejected the compact form with "non-hex characters" -- the `...` looks like garbage to a hex parser. The fix is the resolver's Tier 0: detect `...`, split into head and tail, run a combined prefix-AND-suffix query. Without this, the compact display would silently undermine itself.

### HV.6 — `csb show <unknown>` exits 1 with a clear message

Validates: the no-match exit-code contract — scripts can rely on exit code 1 to mean "no such session."

- [ ] **Steps** (pick your shell):

  **cmd.exe**:
  ```cmd
  csb show deadbeefnotreal0000
  echo Exit code: %ERRORLEVEL%
  ```

  **PowerShell**:
  ```powershell
  csb show deadbeefnotreal0000
  Write-Host "Exit code: $LASTEXITCODE"
  ```

  **POSIX**:
  ```bash
  csb show deadbeefnotreal0000
  echo "Exit code: $?"
  ```

- [ ] **Expected:** stderr: `No session found matching 'deadbeefnotreal0000'`. Exit code: 1.
- [ ] **Why this matters:** Exit codes are the script-integration contract. 1 = no match, 2 = ambiguous or invalid input. Mixing them up would mislead any user trying to build automation on top of csb.

## Section 1: Resolver -- prefix, suffix, and ambiguity

### 1.1 Full UUID exact match still works (backward compatibility)

- [ ] **Steps:**
  1. Copy a full UUID from any session: `csb list -n 1` shows the compact form. To get the full UUID, run `csb show <prefix> | findstr ID:` (cmd) / `csb show <prefix> | Select-String ID:` (PowerShell) / `csb show <prefix> | grep ID:` (POSIX).
  2. Run `csb show <full-uuid>`.
- [ ] **Expected:** Same output as `csb show <prefix>`. Full UUIDs continue to work for all commands; the sugar is additive.
- [ ] **Check:** No deprecation warning, no behavior change for the legacy input form.

### 1.2 Case-insensitive matching

- [ ] **Steps:**
  1. Pick a UUID prefix containing letters, e.g. `7250DDCE` (uppercase).
  2. Run `csb show 7250DDCE`.
- [ ] **Expected:** Resolves the same as the lowercase form.

### 1.3 Hex with embedded hyphens is accepted

- [ ] **Steps:**
  1. Use a UUID middle segment that includes hyphens, e.g. `7250ddce-6d73`.
  2. Run `csb show 7250ddce-6d73`.
- [ ] **Expected:** Resolves correctly. Hyphens are part of the canonical UUID form and should not be rejected as "non-hex."

### 1.4 Non-hex input is rejected

- [ ] **Steps:**
  1. Run `csb show notahex1234`.
- [ ] **Expected:** stderr: `Error: Session ID input 'notahex1234' contains non-hex characters`. Exit code 2.

### 1.5 Real-data ambiguity (if you have multiple sessions with shared prefixes)

This test depends on your DB having at least two sessions whose UUIDs share a 4-char head. On most machines with 100+ sessions this is likely; on small DBs (<20 sessions) it may not be.

- [ ] **Steps:**
  1. Pick a 4-char prefix you suspect might match multiple sessions. To probe: pick the first 4 chars of a UUID from `csb list -n 20` and try `csb show <4-chars>`.
  2. If it resolves uniquely, that prefix has only one match -- pick a different 4-char prefix and try again.
  3. If it shows an "Ambiguous session ID" message, you've found one.

- [ ] **Expected** (on ambiguous match):
  - stderr begins with `Ambiguous session ID '<query>' (N matches):`
  - Each candidate line shows: `<short-UUID>  <name>  (<project>)  started at: <path>`
  - stderr ends with: `Use a longer prefix or the full UUID.`
  - Exit code: 2.
- [ ] **Check:**
  - The compact UUID form (`<head>-...-<tail>`) is used in the candidate list -- not the full UUID, not just the 8-char head.
  - The candidates include enough info (name + project + start-at) for a human to pick the right one and re-run.

### 1.6 Re-running with a longer prefix disambiguates

If you found an ambiguity in 1.5:

- [ ] **Steps:**
  1. From the candidate list, copy the first 6-8 chars of one specific session's UUID (longer than your original 4 chars).
  2. Run `csb show <longer-prefix>`.
- [ ] **Expected:** Resolves to that one specific session. No ambiguity error.

### 1.7 Pure-suffix match when prefix tier finds nothing

- [ ] **Steps:**
  1. Pick the last 8 chars of any session's UUID -- the tail. Make sure no session's UUID *starts* with those same 8 chars (very likely true on real data).
  2. Run `csb show <tail-8-chars>`.
- [ ] **Expected:** The resolver finds it via suffix match. The session displays normally.

## Section 2: Display defaults and the `--shortid` opt-in

### 2.1 `csb list` `id:` field shows the FULL UUID by default

- [ ] **Steps:**
  1. Run `csb list -n 5`.
- [ ] **Expected:** Every `id:` line shows the full 36-char UUID.
- [ ] **Why this matters:** Users copying from the output to paste into `claude --resume <uuid>` need the complete form (claude has no short-UUID resolver). Defaulting to the compact form would break this workflow.

### 2.2 `csb search` session header shows the FULL UUID by default

- [ ] **Steps:**
  1. Run `csb search the --limit 3 --no-color`. (`the` is a common-enough word to find hits on any dev machine.)
- [ ] **Expected:** Each per-session header line shows `<name>  <full-UUID>  (<project>)  last: <ts>`.
- [ ] **Why this matters:** Uniform with `csb list`. The same copy-paste-into-native-tool flow applies after content search.

### 2.3 `--shortid` (also `-sid`) opts in to the compact form

- [ ] **Steps:**
  1. `csb list -n 3 --shortid`
  2. `csb scan --shortid` (output may be empty if cwd has no sessions; that's fine)
  3. `csb search the --limit 3 --shortid --no-color`
- [ ] **Expected:** All three show `<head>-...-<tail>` in their UUID columns / id lines. Same flag works across all three commands.

### 2.4 `csb show` always displays the FULL UUID (no flag)

This is a deliberate design choice -- `csb show` is the detail-view command and always shows the complete UUID for paste-into-anything use.

- [ ] **Steps:**
  1. Run `csb show <any-prefix>`.
- [ ] **Expected:** The `ID:` line shows the FULL 36-char UUID. No `--shortid` flag on `csb show`.
- [ ] **Why this matters:** A detail view that hides part of the data would be self-defeating; users went to `csb show` precisely to see everything.

### 2.5 `csb show` output is Rich-colorized when Rich is available

- [ ] **Steps:**
  1. Run `csb show <any-prefix>` in a terminal that supports ANSI color (modern Windows Terminal, PowerShell 7, any Unix terminal).
- [ ] **Expected:** Session name in **bold cyan**, start folder in **bold green**, `Resume:` and any `Restore with:` hints in **bold yellow**, deleted markers (if applicable) in **red**, field labels and timestamps **dim**.
- [ ] **Note:** if Rich isn't installed (`pip install rich`), output falls back to plain text with the same content -- no errors.

### 2.6 Visual readability sanity check

Open a wide terminal (>=120 cols) and run:

- [ ] **Steps:**
  1. `csb list -n 10`
  2. `csb list -n 10 --shortid`
  3. `csb search the --limit 5 --no-color`
  4. `csb search the --limit 5 --no-color --shortid`
- [ ] **Check:**
  - Default outputs (full UUID) align cleanly across all rows -- the fixed-width UUID column makes scanning easy.
  - `--shortid` outputs are noticeably more compact horizontally. Useful when terminal width is tight.
  - No alignment artifacts in either mode.
  - The compact `<head>-...-<tail>` doesn't look broken with varied UUID patterns (lots of leading zeros, all-letter prefixes).

## Section 3: Migrated commands -- show / resume / restore use the resolver

### 3.1 `csb resume <prefix>` (without launching)

Validates: `csb resume` accepts short prefixes. (Section 5 covers actually launching claude.)

- [ ] **Steps:**
  1. Pick a session prefix you know exists.
  2. **DO NOT actually launch claude** if you don't want to switch session context. Use `--dry-run` if available, OR be ready to immediately Ctrl-C out of the new claude session.

  Alternative non-launching verification (safer):

  **cmd.exe**:
  ```cmd
  csb resume nosuchidthatwillfail
  echo Exit code: %ERRORLEVEL%
  ```

  **PowerShell**:
  ```powershell
  csb resume nosuchidthatwillfail
  Write-Host "Exit code: $LASTEXITCODE"
  ```

  **POSIX**:
  ```bash
  csb resume nosuchidthatwillfail
  echo "Exit code: $?"
  ```

- [ ] **Expected:** stderr: `No session found matching 'nosuchidthatwillfail'`. Exit code: 1. No `claude` process launches.

### 3.2 `csb restore <prefix>` accepts short prefixes

If you don't have any deleted sessions, this test is the no-match path.

- [ ] **Steps:**
  1. Run `csb restore nosuchprefix1234`.
- [ ] **Expected:** stderr: `No session found matching 'nosuchprefix1234'`. Exit code: 1.

If you DO have a deleted session, you can additionally run `csb restore <real-prefix> --dry-run` to confirm the resolver works without performing the actual restore.

### 3.3 Ambiguous prefix returns exit 2 from all three commands

If Section 1.5 found you an ambiguous prefix:

- [ ] **Steps:**
  1. Run `csb show <ambiguous-prefix>; echo Exit: $LASTEXITCODE` (PowerShell) or equivalent.
  2. Run `csb resume <ambiguous-prefix>`.
  3. Run `csb restore <ambiguous-prefix>`.
- [ ] **Expected:** All three print the same candidate list (or very similar). All three exit with code 2.

## Section 4: Error-message quality and clarity

These are subjective UX checks — mocks can't tell you whether the wording is helpful.

### 4.1 Empty input error message

- [ ] **Steps:** Run `csb show ""` (empty string).
- [ ] **Expected:** A clear error message naming the problem.
- [ ] **Check:** Does the message say "Empty session ID input" or similar? Is it obvious what to do next?

### 4.2 Too-short input error message names the minimum

- [ ] **Steps:** Run `csb show xy`.
- [ ] **Expected:** Message mentions "at least 4 hex chars" so the user knows the threshold.

### 4.3 Non-hex input error names the constraint

- [ ] **Steps:** Run `csb show xyz!1234`.
- [ ] **Expected:** Message mentions "non-hex" so the user knows to remove special characters.

### 4.4 Ambiguous candidate list is informative

If Section 1.5 found an ambiguity:

- [ ] **Check:**
  - The candidate list shows enough info that a human can pick the right one: name, project, start-at.
  - If there are more than 10 candidates, the list truncates with `... and N more`.
  - The closing hint says to use a longer prefix.

## Section 5: The full search-to-resume workflow (the actual user story)

This is the test that validates "user found a session via search and wants to load into it." End-to-end real workflow.

**Skip this section** if you don't want to switch your active Claude Code session.

### 5.1 Search → identify → resume

- [ ] **Steps:**
  1. Run `csb search <a-phrase-you-recall-from-a-past-session> --limit 5`. Pick something you know exists, like a project name or technical term.
  2. Read the session headers. Each shows the compact UUID form.
  3. Pick the session you want to resume. Copy its 8-char head.
  4. Run `csb resume <8-char-head>`.
- [ ] **Expected:**
  - `claude --resume <full-uuid>` launches with `cwd=<start-at>`.
  - You enter the chosen session.
  - The compact UUID was sufficient -- no need to copy the full 36-char form.
- [ ] **Why this matters:** This is the entire point of the v0.2.7 sugar -- removing the friction between "I found the session" and "I'm in the session." If this flow feels awkward, the sugar isn't working in practice.

### 5.2 Cleanup

After the resumed claude session exits (Ctrl+D / `/exit`), control returns to your shell with the right exit code. Verify with:

**cmd.exe**:
```cmd
echo Exit code: %ERRORLEVEL%
```

**PowerShell**:
```powershell
Write-Host "Exit code: $LASTEXITCODE"
```

**POSIX**:
```bash
echo "Exit code: $?"
```

- [ ] **Expected:** Exit code matches whatever claude returned (typically 0).

## Section 6: Cross-shell sanity check

Pick the OTHER shell you don't usually use and run HV.1 + HV.4 + HV.6 again.

- [ ] **Steps:** Run the three HV tests in your non-default shell (e.g. PowerShell if you usually use cmd, or bash if you usually use PowerShell).
- [ ] **Check:**
  - Output renders cleanly (no broken ANSI sequences, no missing characters from codepage issues).
  - Exit codes are reported correctly via the shell's variable (`%ERRORLEVEL%`, `$LASTEXITCODE`, `$?`).
  - The compact UUID format displays the same in all shells.

## What the automated tests DON'T cover

Explicit gaps that ONLY manual testing can verify:

- **Real-data collision behavior** — does the candidate list actually show informative metadata for sessions on the user's machine? Mocks use synthetic rows.
- **Visual readability of the compact UUID form** — does `<head>-...-<tail>` scan well in real terminal output, with realistic UUID values (lots of leading zeros, all-letter prefixes, etc.)? Mocks can't evaluate "feels good to read."
- **Cross-shell output rendering** — Windows cp437 / cp1252 codepage handling, PowerShell's ANSI quirks, bash on Windows (Git Bash, WSL). Mocks run inside Python's stdout.
- **End-to-end search-to-resume flow** — does the sugar actually save typing in practice? Only a human running the real workflow knows.
- **Exit-code contract via shell variables** — `%ERRORLEVEL%`, `$LASTEXITCODE`, `$?` all behave slightly differently. Mocks return integers, not shell variables.
- **`claude --resume` integration after resolver** — does the full UUID from the resolver actually launch claude correctly? `cmd_resume` mocks `subprocess.run` so the launch is never exercised against the real claude binary.
- **Error message clarity** — "is this helpful?" is a human judgment.
- **Edge cases with the user's actual DB** — sessions with deleted/null fields, very long session names, projects with special chars, etc.

## Reporting issues found

1. Note the section and step number (e.g., "HV.1 step 2").
2. Capture the exact command and its full output:

   **cmd.exe**:
   ```cmd
   csb show 7250ddce > %TEMP%\bug_output.txt 2>&1
   type %TEMP%\bug_output.txt
   ```

   **PowerShell**:
   ```powershell
   csb show 7250ddce *> $env:TEMP\bug_output.txt
   Get-Content $env:TEMP\bug_output.txt
   ```

   **POSIX**:
   ```bash
   csb show 7250ddce > /tmp/bug_output.txt 2>&1
   cat /tmp/bug_output.txt
   ```

3. Note `csb --version`, your OS, shell, and Python version (`python --version`).
4. File a GitHub issue at https://github.com/DazzleML/Claude-Session-Backup/issues referencing this checklist and the version tag.
