# Human Test Checklist: Durable SessionEnd backup, detached + no window (v0.3.8)

**Target version:** v0.3.8
**Ships with commit:** v0.3.8: fix: durable SessionEnd backup (detached child survives teardown) + no console window
**Companion automated tests:** 26 in `tests/test_backup_hook.py` + `tests/test_hook.py`. Total 624/624 pass.

## What this is

v0.3.7 backed up in the background but with no detach flags, so Claude Code's process-tree teardown hard-killed the SessionEnd backup mid-run. v0.3.8 spawns the backup **decoupled** from the session's process tree (Windows: `CREATE_NO_WINDOW | CREATE_NEW_PROCESS_GROUP`; POSIX: `start_new_session=True`) so it **survives teardown and completes**, and shows **no console window**. Mocks can't fire real hooks or exercise teardown, so this checklist verifies the end-to-end behavior by hand.

## Prerequisites

1. csb v0.3.8 installed (`pip install -e ".[dev]"`).
2. The plugin is **directory-sourced** at the repo, so the updated hook is live on the next session -- **no `claude plugin update` needed; just restart Claude Code.** (`hooks.json` registrations are unchanged from v0.3.7.)
3. A `~/.claude` git repo with sessions so `csb backup` has real work to do.

## High-Value Verification (~5 min)

### HV.1 -- No console window on exit OR start (Windows)
- [ ] Start a Claude Code session, do a little work, then `/exit`. Start a new session.
- [ ] **Expected:** **no console/terminal window flashes** on either the exit or the start. (`CREATE_NO_WINDOW` gives `csb` a hidden console that its `git` children inherit.)
- [ ] **Why it matters:** the prior flag (`DETACHED_PROCESS`) made each `git` child pop its own console -- annoying and the original report.

### HV.2 -- SessionEnd backup actually COMPLETES after teardown
- [ ] In session A, say a unique phrase. `/exit`. **Do NOT start another session yet.**
- [ ] Wait ~20s, then (PowerShell): `Get-Content "$env:USERPROFILE\.claude\csb-logs\backup-SessionEnd.log" -Tail 3`
- [ ] `csb search "<unique phrase>"` finds session A, and `csb show <A-uuid>` is fresh -- **without** having started a catch-up session.
- [ ] **Why it matters:** this is the core fix. In v0.3.7 the just-ended session stayed un-indexed until the next SessionStart; now SessionEnd finishes on its own.

### HV.3 -- No stale lock left behind
- [ ] After the exit in HV.2: `Test-Path "$env:USERPROFILE\.claude\.csb-backup.lock"`
- [ ] **Expected:** the backup released its lock cleanly (no lock, or a fresh one that the next run reclaims). No permanent stale lock.

### HV.4 -- Concurrent SessionEnd + SessionStart serialize safely
- [ ] `/exit` and immediately relaunch. The exiting session's SessionEnd backup and the new session's SessionStart backup may overlap.
- [ ] **Expected:** one acquires the lock and runs; the other **skips fast** (logged) -- no double commit, no corruption. Check both `backup-SessionEnd.log` and `backup-SessionStart.log`.

## Section 1: spawn flags (cross-platform, no real backup)

### 1.1 detach kwargs per platform
- [ ] **Steps** (same across shells):
  ```
  python -c "import importlib.util,pathlib; s=importlib.util.spec_from_file_location('bh',pathlib.Path('hooks/scripts/backup-hook.py')); m=importlib.util.module_from_spec(s); s.loader.exec_module(m); m.sys.platform='win32'; print('win', hex(m._detach_kwargs()['creationflags'])); m.sys.platform='linux'; print('posix', m._detach_kwargs())"
  ```
- [ ] **Expected:** `win 0x8000200` (CREATE_NO_WINDOW | CREATE_NEW_PROCESS_GROUP) and `posix {'start_new_session': True}`.
