## Prerequisites

1. **OS / Python**: Windows (primary; macOS / Linux secondary). Python 3.10+ with the `claude_session_backup` package installed (`pip install -e .` from the repo root, or `pip install git+https://github.com/DazzleML/Claude-Session-Backup.git@<commit>`).
2. **Real claude-code installation**: `claude` must be in `PATH` for Section 11.1's success-path test (the actual resume into a session). Sections 11.2-11.4 work without `claude`.
3. **Recent SQLite index**: run `csb backup --no-commit` once before starting if your index is stale. Section 12 reads against the real `~/.claude/session-backup.db`.
4. **A few real sessions in your index**: ideally with at least one whose `start_folder` differs from your current terminal cwd (Section 11) and ideally one where the slug-decoded path differs from `start_folder` (Section 11.4 / 12.2 -- rare; AMD_INTIGRITI-style cases).
5. **Test isolation** (optional, only if you want to avoid touching your real index):

   **cmd.exe**:
   ```cmd
   set CLAUDE_SESSION_BACKUP_DB=%TEMP%\csb-checklist-sandbox.db
   csb backup --no-commit
   ```

   **PowerShell**:
   ```powershell
   $env:CLAUDE_SESSION_BACKUP_DB = "$env:TEMP\csb-checklist-sandbox.db"
   csb backup --no-commit
   ```

   **POSIX / WSL**:
   ```bash
   export CLAUDE_SESSION_BACKUP_DB=/tmp/csb-checklist-sandbox.db
   csb backup --no-commit
   ```

   Most checklist tests are read-only against the index, so isolation is optional. The `csb resume` tests (Section 11) are not affected by index location -- they exec `claude` directly -- but the index is consulted to look up the session.

**Reset between sections**: each section is independent. No global state changes between them. Section 11.2 (deleted-folder repro) is the only test that requires creating + deleting a test folder; instructions are inline.

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. They cover the highest-impact user-visible behavior. If any of them fail, the work is broken and the detailed sections below probably can't save you.

### HV.1 -- `csb resume <uuid>` from a foreign cwd actually works AND accepts keystrokes (#24 alpha-blocker)

Validates: the original repro for issue #24 is fixed. `cmd_resume` now launches `claude --resume` via `subprocess.run` with `cwd=target` (the slug-decoded path). The subprocess approach is required on Windows because Python's `os.execvp` there is `_spawnv(P_OVERLAY, ...)` and the controlling-TTY relationship doesn't transfer cleanly.

- [ ] **Steps:**
  1. From your terminal, `cd` to a directory that is NOT any session's `start_folder`. The repo root works fine.
  2. Pick a session UUID prefix from `csb list` whose `start_folder` is different from your current cwd.
  3. Run `csb resume <uuid-prefix>`.
- [ ] **Expected:**
  1. `csb` prints the resume preamble (`Resuming: ...`, `cd <path>`, `claude --resume <uuid>`)
  2. `claude` launches in the slug-decoded directory and successfully resumes the session. **NOT** "No conversation found with session ID: ..."
  3. The Claude TUI **accepts keystrokes**. Type something into the resumed session and confirm the input shows up. This is the third regression we caught -- the prior `os.execvp` implementation broke the TTY handoff on Windows; the subprocess.run implementation must work.
  4. After exiting claude (Ctrl+D / `/exit`), control returns to the original csb-parent shell; csb propagates claude's exit code.
- [ ] **Why this matters:** This is the alpha-blocker, and it has three failure modes that have all been addressed: (a) `csb resume` not actually `cd`-ing (Layer 1); (b) `csb resume` `cd`-ing to `first_cwd` instead of slug-decoded path (Layer 2); (c) `os.execvp` breaking stdin on Windows (subprocess.run fix). All three need a human to verify -- automated mocks can verify (a) and (b) but only a human typing into a real terminal can verify (c).

### HV.2 -- `csb scan amdead` works as broad metadata search (#20 core fix)

Validates: the bare positional is no longer auto-resolved against cwd; it's a metadata search.

- [ ] **Steps:**
  1. From a directory that does NOT contain an `amdead/` subfolder, run `csb scan amdead`.
- [ ] **Expected:** Sessions whose name, project, `start_folder`, or top-N `folder_usage` paths contain "amdead". No `[info]` hint (cwd has no `amdead` subfolder).
- [ ] **Why this matters:** Pre-fix, this would have silently treated `amdead` as `<cwd>\amdead` and returned nothing useful. The user's mental model -- "find sessions that mention amdead" -- now matches behavior.

### HV.3 -- `csb scan -d amdead` is path-strict (#20 explicit mode)

Validates: the new `-d` flag exists and resolves the pattern against the actual filesystem.

- [ ] **Steps:**
  1. From a directory that contains an `amdead/` subfolder, run `csb scan -d amdead`.
- [ ] **Expected:** Sessions whose `start_folder` OR top-N `folder_usage` paths equal `<cwd>\amdead` or are descendants. No sibling-prefix matches (e.g., `amdead-fork` should NOT appear unless you used `-d amdead*`).
- [ ] **Why this matters:** Path-strict mode is now reachable explicitly. Without `-d`, the user has no way to express "search just this folder" anymore.

### HV.4 -- `csb scan ./amdead` shortcut auto-promotes to `-d`

Validates: ergonomic shortcut works without typing the flag name.

- [ ] **Steps:**
  1. From a directory containing an `amdead/` subfolder, run `csb scan ./amdead`.
- [ ] **Expected:** Same output as `csb scan -d amdead`. No flag-name needed.
- [ ] **Why this matters:** The shortcut is what users will reach for in practice; if it doesn't work, the rest of the `-d` machinery is hard to discover.

### HV.5 -- `csb list` "start at" lines stable; no false `<unresolved:>` (#23 + #19 no-regression)

Validates: pathkit disambiguation didn't break #19's existing slug decoder.

- [ ] **Steps:**
  1. Run `csb list -n 20`.
  2. Visually scan the "start at" lines.
- [ ] **Expected:** Sessions whose original cwd is still on disk show real paths (`C:\code\amdead`, `C:\Users\Extreme`, etc.). Only sessions whose original cwd has been deleted should show `<unresolved:slug>`. No silently-wrong paths.
- [ ] **Why this matters:** #23 refactored the slug-decoder. If it broke #19's existing behavior, EVERY session's "start at" line could be wrong, and `csb resume`'s Layer 2 fix (which depends on the slug-decoded path) would be unreliable.

### HV.6 -- Full automated test suite passes

Validates: regressions across all 221 automated tests.

- [ ] **Steps:**
  1. From the repo root: `python -m pytest -q`.
- [ ] **Expected:** `221 passed` (or more if tests have been added since the checklist was written). Zero failures.
- [ ] **Why this matters:** Cheap insurance against any subtle import-time or fixture issue. If this fails, stop and investigate before running the human sections.

  - **Expected**: same output as `csb scan -d .`. Session list under cwd, sorted by last activity.
  - **Verifies**: bare scan default unchanged from pre-v0.2.3.

- [ ] **1.3** `csb scan -NU` from same dir.

## Section 2a -- `./` and `.\` shortcut (auto-promote to `-d`)

- [ ] **2a.1** `csb scan ./amdead` from a directory containing `amdead/`.
  - **Expected**: same output as `csb scan -d amdead`. No `-d` flag needed.
- [ ] **2a.2** `csb scan .\amdead` (Windows backslash variant).
  - **Expected**: same as 2a.1.
- [ ] **2a.3** `csb scan .` (bare dot).
  - **Expected**: same as `csb scan -d .` and `csb scan` (bare).
- [ ] **2a.4** `csb scan ./amdead my-paper` (path shortcut + term filter).
  - **Expected**: same as `csb scan -d amdead my-paper` (combined scope + filter).
- [ ] **2a.5** `csb scan -d redditslack ./amdead` (explicit -d + dot-prefix as positional).
  - **Expected**: explicit `-d` wins; `./amdead` is treated as the term (no auto-promote when -d/-D is already explicit).
- [ ] **2a.6** `csb scan .gitignore` (dotfile name, no slash).
  - **Expected**: NOT promoted -- treated as broad term search for `.gitignore`. The shortcut requires `./` or `.\` prefix.

## Section 2 -- Term-only mode (NEW broad metadata search)

- [ ] **2.1** `csb scan AMD_INTIGRITI` from any directory.
  - **Expected**: session with that name in the title appears (regardless of cwd).
  - **Verifies**: term matches against `session_name`.
- [ ] **2.2** `csb scan amdead` from a directory that does NOT contain a folder named `amdead`.
  - **Expected**: results include sessions whose folder_usage paths contain "amdead". No `[info]` hint (term doesn't coincide with a cwd subfolder).
- [ ] **2.3** `csb scan amdead` from a directory that DOES contain a folder named `amdead`.
  - **Expected**: results AND a `[info]` hint on stderr: `[info] 'amdead' is also a folder under cwd. Use 'csb scan -d amdead' for path-strict search.`
  - **Verifies**: hint emission, stderr routing.
- [ ] **2.4** Suppress stderr to confirm the hint truly goes to stderr (not stdout).

  **cmd.exe**:
  ```cmd
  csb scan amdead 2>nul
  ```

  **PowerShell**:
  ```powershell
  csb scan amdead 2>$null
  ```

  **POSIX / WSL**:
  ```bash
  csb scan amdead 2>/dev/null
  ```

  - **Expected:** hint suppressed; results visible on stdout. If the hint appears, it's contaminating stdout -- bug.
  - **Verifies:** hint goes to stderr, not stdout.
- [ ] **2.5** `csb scan AMDEAD` (uppercase) -- compare to `csb scan amdead`.
  - **Expected**: same result set (case-insensitive).

## Section 3 -- `-d` (path-strict, with descendants)

- [ ] **3.1** `csb scan -d amdead` from a directory containing an `amdead` subfolder.
  - **Expected**: sessions whose start_folder OR top-N folder_usage paths are `<cwd>\amdead` or descendants. No sibling-prefix matches (e.g., `amdead-fork` should NOT be included).
- [ ] **3.2** `csb scan -d <absolute-path>` (e.g., `csb scan -d C:\code\amdead`).
  - **Expected**: same as 3.1 but with explicit absolute path; cwd doesn't matter.
- [ ] **3.3** `csb scan -d amdead*` (wildcard) from a directory containing both `amdead` and `amdead-fork`.
  - **Expected**: sessions matching either folder, including descendants.
  - **Verifies**: trailing-wildcard SQL LIKE pattern.
- [ ] **3.4** `csb scan -d /nonexistent-dir` (using a path you know doesn't exist).
  - **Expected**: `[warning]` on stderr; if no term given, no results returned (silently empty).
  - **Verifies**: warning emission for missing path.
- [ ] **3.5** `csb scan -d /nonexistent-dir amdead`.
  - **Expected**: `[warning]` on stderr, then results from broad-term search for "amdead".
  - **Verifies**: graceful fallback to term-mode when -d resolves to nothing.

## Section 4 -- `-D` (path-strict, exact folder only)

- [ ] **4.1** `csb scan -D amdead` from a directory containing `amdead/` with subdirs.
  - **Expected**: sessions whose start_folder OR top-N folder_usage paths EQUAL `<cwd>\amdead`. Sessions whose folder_usage points only at `<cwd>\amdead\sub` should NOT match (descendants excluded).
- [ ] **4.2** `csb scan -D amdead*` (wildcard with no descendants).
  - **Expected**: sessions matching `amdead`, `amdead-fork`, etc. -- but NOT their descendants like `amdead\sub` or `amdead-fork\sub`.
  - **Verifies**: NOT LIKE clause excludes descendants while LIKE allows siblings.

## Section 4b -- `-s` (start_folder only; "what originated here?")

- [ ] **4b.1** `csb scan -s C:\code` -- a directory many sessions start in.
  - **Expected**: only sessions whose `start_folder` is `C:\code` or a descendant. Sessions where `C:\code` appears only in `folder_usage` (e.g., a session that started elsewhere but did work under `C:\code`) should NOT match.
  - **Verifies**: `-s` skips folder_usage entirely.
- [ ] **4b.2** Compare: `csb scan -d C:\code -n 50` and `csb scan -s C:\code -n 50`.
  - **Expected**: `-d` returns a superset of `-s` (it matches via start_folder OR folder_usage; `-s` matches start_folder only).
- [ ] **4b.3** `csb scan -s amdead*` (wildcard).
  - **Expected**: sessions whose start_folder begins with `<cwd>\amdead` (sibling-prefix expansion via trailing `*`).
- [ ] **4b.4** `csb scan -s . ` (dot, expands to cwd).
  - **Expected**: sessions whose start_folder is cwd or a descendant. Useful as "is there anything I can resume that started in this folder?"

## Section 5 -- Combined: `-d`/`-D`/`-s` + term

- [ ] **5.3** `csb scan -s amdead my-paper`.

## Section 11 -- `csb resume` launches claude correctly with TTY handoff (#24)

The repro for #24 was: from a foreign cwd, `csb resume <uuid>` printed `cd <path>` as informational text and then exec'd `claude --resume <uuid>` without changing directory, so claude failed with "No conversation found." A second-order failure surfaced after the initial fix landed: even when `os.chdir` was added before `os.execvp`, the Windows TTY handoff was broken (claude TUI rendered but stdin keystrokes went into the void). The current implementation uses `subprocess.run(..., cwd=target)` to avoid both failure modes.

- [ ] **11.1** Pick a session whose `start_folder` is different from your current shell cwd. From a different directory than `start_folder`, run `csb resume <session-id-prefix>`.
  - **Expected**: `csb` prints the resume preamble (`Resuming: ...`, `cd <path>`, `claude --resume <uuid>`), then `claude` launches in the right cwd and successfully resumes. The Claude TUI **accepts keystrokes** (this verifies the subprocess.run-based fix, not just the cwd resolution). After you `/exit` claude, control returns to the original shell with claude's exit code.
- [ ] **11.2** Pick a session whose recorded `start_folder` no longer exists on disk (delete a test folder first if needed; or pick one with an unresolved `start at`).
  - **Expected**: `csb` invokes `subprocess.run(..., cwd=<deleted-path>)`, which raises `FileNotFoundError`. `csb` prints `Error: cannot cd to <path>: ...` on stderr and exits with code 1. `claude` is NOT launched. The user can copy the printed UUID and resume manually.
- [ ] **11.3** Verify exit codes:
  - Successful resume: rc = whatever claude exits with (typically 0).
  - Target folder deleted: rc 1.
  - `claude` not in PATH: rc 1 (with "Error: 'claude' command not found in PATH" message; disambiguated from the deleted-target case via `os.path.isdir(target)` check inside the `FileNotFoundError` handler).
- [ ] **11.4** Verify the slug-decoded path is preferred over `start_folder`. If a session has `jsonl_path` set in the index AND its slug-decoded path differs from `start_folder` (this is rare but happens for AMD_INTIGRITI-style sessions), confirm the `cd` line and the actual subprocess cwd both show the slug-decoded value.
  - Easiest way to check: run `csb show <session-id>` and compare the "start at" line to the `csb resume` `cd` line -- they should match.

## Section 12 -- pathkit multi-candidate slug disambiguation (#23)

Most installations don't have a genuinely-ambiguous slug on disk (it requires literally creating two real folders that sanitize to the same slug, e.g., `New--Project` and `New/.Project` siblings). This section is therefore primarily covered by automated tests (`tests/test_pathkit.py`); the human verification is light.

- [ ] **12.1** Run `csb list` and visually inspect the "start at" lines on a few sessions. Confirm none show `<unresolved:slug>` for sessions whose original cwd is still on disk. Sessions whose original cwd has been deleted *should* show `<unresolved:>` and not silently pick a wrong path.
- [ ] **12.2** (Optional, only if you actually have ambiguous slugs) Identify a session whose slug decodes to multiple real folders. Verify the "start at" line picks the folder that matches the session's `first_cwd` (per the SQLite `start_folder` column).
- [ ] **12.3** No-regression check: confirm previously-correct sessions (#19's verification cases) still resolve correctly. Pick AMD_INTIGRITI or any other session with a non-trivial slug; confirm "start at" matches what it showed pre-#23.
- [ ] **12.4** `csb resume` chain integration: `csb resume <uuid>` should now `cd` to the slug-decoded path, not `start_folder` -- verified in Section 11.4 above.

## What the automated tests DON'T cover

This checklist exists because automated tests can't catch:

1. **Terminal rendering quirks**: Rich-vs-plain output, color, alignment, line wrapping in narrow terminals.
2. **Stderr/stdout split**: hint and warning lines must NOT contaminate stdout when piping or redirecting (Sections 2.4, 3.4, 3.5).
3. **Real subprocess invocation**: parser tests use `parser.parse_args()` directly. End-to-end CLI invocation could fail on Python entry-point resolution or import-time side effects.
4. **Real `claude --resume` exec into a session** (Section 11.1): `cmd_resume` tests mock `os.execvp` -- only a human running the checklist verifies the actual launch into `claude` works end-to-end.
5. **Cross-shell behavior**: PowerShell, cmd.exe, WSL bash all handle quoting and path separators differently. Section 9.4 verifies the scan command across shells; Section 11 only on the user's primary shell.
6. **Real SQLite + real index**: SQL tests use temp-fixture indexes. This checklist exercises the actual `~/.claude/session-backup.db` (or the sandbox path from Prerequisites) against real session data.
7. **Real filesystem walk in `~/.claude/projects/`** (Section 12): pathkit unit tests use `tmp_path`; only a checklist run hits the real layout with all its Windows path quirks (drive letters, mixed casing, network drives if mapped).
8. **Discoverability**: does `csb scan --help` clearly distinguish `-d` vs `-D` vs `-s` vs term? Read it from a fresh-eyes perspective.
9. **Refactor regression**: `cmd_scan` and `cmd_resume` were substantially rewritten; pathkit gained new functions and refactored `_decode_under` into a thin wrapper. Confirm `csb backup`, `csb list`, `csb show`, `csb status` still work normally (sanity).
10. **OSError variations on Windows**: `os.chdir` failure modes vary across "drive not mounted" / "permission denied" / "path too long" / "deleted concurrently". Section 11.2 covers the common one (deleted folder); other OSError shapes are mocked uniformly.

## Reporting issues found

1. Note the section and step number (e.g., "11.4 fails: cd line shows `start_folder`, not the slug-decoded path").
2. Capture the exact command and the full output (stdout + stderr separately if possible).
3. Note your platform: OS, Python version, csb version (`csb --version`), shell (cmd.exe / PowerShell / bash).

## Cleanup

This checklist creates no persistent test data unless you ran Section 11.2's optional folder-delete repro. To restore:

1. **If you set the `CLAUDE_SESSION_BACKUP_DB` sandbox env var in Prerequisites step 5**: unset it for your shell session (or close the shell). Optionally delete the sandbox file:

   **cmd.exe**:
   ```cmd
   if exist %TEMP%\csb-checklist-sandbox.db del %TEMP%\csb-checklist-sandbox.db
   set CLAUDE_SESSION_BACKUP_DB=
   ```

   **PowerShell**:
   ```powershell
   if (Test-Path "$env:TEMP\csb-checklist-sandbox.db") { Remove-Item "$env:TEMP\csb-checklist-sandbox.db" }
   Remove-Item Env:\CLAUDE_SESSION_BACKUP_DB -ErrorAction SilentlyContinue
   ```

   **POSIX / WSL**:
   ```bash
   rm -f /tmp/csb-checklist-sandbox.db
   unset CLAUDE_SESSION_BACKUP_DB
   ```

2. **If you ran `csb scan -d /nonexistent` and noticed any unexpected entries in your real index** (the warning path is NOT supposed to write anywhere -- if it did, that's a bug to file): inspect via `csb status` and re-run `csb backup --no-commit` to refresh.

3. **If you ran Section 11.2's deleted-folder repro**: that test creates a temp folder and deletes it before running `csb resume` against a session whose `start_folder` matched the deleted path. The session in the index still references the (now-deleted) path; this is the expected state for the bug repro and doesn't need cleanup.
