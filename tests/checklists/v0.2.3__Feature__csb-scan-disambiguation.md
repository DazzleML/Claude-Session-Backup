# csb v0.2.3 -- `scan` term-vs-folder disambiguation (Feature)

Companion to the automated tests in `tests/test_cli.py`, `tests/test_index.py`, and `tests/test_commands.py`. The automated tests cover parser behavior, SQL helper logic, and pure path-resolution helpers; this checklist covers the human-noticeable surfaces that mocks systematically miss: terminal rendering, real subprocess behavior, hint emission to stderr, `--json` cleanliness, and how the new flags compose with existing scan output.

**Scope of this release**: closes #20. Ships alongside #19 (already closed) + #21 (already merged) + display_top_folders follow-up + #23 (deferred).

**Prerequisites**:
- csb installed locally with the new `cmd_scan` (commit `<TBD>` from this work).
- A SQLite index that's been recently updated (`csb backup` if stale).
- Real session data covering at least one folder under cwd that's also a meaningful project name.

**Reset between sections**: each section is independent; no global state needs resetting.

---

## Section 1 -- Bare scan (no breaking change to ergonomic default)

- [ ] **1.1** From a project directory (e.g., `C:\code\amdead`), run `csb scan`.
  - **Expected**: same output as `csb scan -d .`. Session list under cwd, sorted by last activity.
  - **Verifies**: bare scan default unchanged from pre-v0.2.3.
- [ ] **1.2** `csb scan -n 5` from same dir.
  - **Expected**: top 5 results.
- [ ] **1.3** `csb scan -NU` from same dir.
  - **Expected**: only sessions whose start_folder matches; folder_usage step skipped.

## Section 2a -- `./` and `.\` shortcut (auto-promote to `-d`)

- [ ] **2a.1** `csb scan ./amdead` from a directory containing `amdead/`.
  - **Expected**: same output as `csb scan -d amdead`. No `-d` flag needed.
- [ ] **2a.2** `csb scan .\amdead` (Windows backslash variant).
  - **Expected**: same as 2a.1.
- [ ] **2a.3** `csb scan .` (bare dot).
  - **Expected**: same as `csb scan -d .` and `csb scan` (bare).
- [ ] **2a.4** `csb scan ./amdead my-paper` (path shortcut + term filter).
  - **Expected**: same as `csb scan -d amdead my-paper` (combined scope + filter).
- [ ] **2a.5** `csb scan -d redditslack ./amdead` (explicit -d + dot-prefix as positional).
  - **Expected**: explicit `-d` wins; `./amdead` is treated as the term (no auto-promote when -d/-D is already explicit).
- [ ] **2a.6** `csb scan .gitignore` (dotfile name, no slash).
  - **Expected**: NOT promoted -- treated as broad term search for `.gitignore`. The shortcut requires `./` or `.\` prefix.

## Section 2 -- Term-only mode (NEW broad metadata search)

- [ ] **2.1** `csb scan AMD_INTIGRITI` from any directory.
  - **Expected**: session with that name in the title appears (regardless of cwd).
  - **Verifies**: term matches against `session_name`.
- [ ] **2.2** `csb scan amdead` from a directory that does NOT contain a folder named `amdead`.
  - **Expected**: results include sessions whose folder_usage paths contain "amdead". No `[info]` hint (term doesn't coincide with a cwd subfolder).
- [ ] **2.3** `csb scan amdead` from a directory that DOES contain a folder named `amdead`.
  - **Expected**: results AND a `[info]` hint on stderr: `[info] 'amdead' is also a folder under cwd. Use 'csb scan -d amdead' for path-strict search.`
  - **Verifies**: hint emission, stderr routing.
- [ ] **2.4** `csb scan amdead 2>/dev/null` (suppress stderr) from same dir.
  - **Expected**: hint suppressed; results visible on stdout. (Linux/WSL; on Windows PowerShell use `2>$null`.)
  - **Verifies**: hint truly goes to stderr, not stdout.
- [ ] **2.5** `csb scan AMDEAD` (uppercase) -- compare to `csb scan amdead`.
  - **Expected**: same result set (case-insensitive).

## Section 3 -- `-d` (path-strict, with descendants)

- [ ] **3.1** `csb scan -d amdead` from a directory containing an `amdead` subfolder.
  - **Expected**: sessions whose start_folder OR top-N folder_usage paths are `<cwd>\amdead` or descendants. No sibling-prefix matches (e.g., `amdead-fork` should NOT be included).
- [ ] **3.2** `csb scan -d <absolute-path>` (e.g., `csb scan -d C:\code\amdead`).
  - **Expected**: same as 3.1 but with explicit absolute path; cwd doesn't matter.
- [ ] **3.3** `csb scan -d amdead*` (wildcard) from a directory containing both `amdead` and `amdead-fork`.
  - **Expected**: sessions matching either folder, including descendants.
  - **Verifies**: trailing-wildcard SQL LIKE pattern.
- [ ] **3.4** `csb scan -d /nonexistent-dir` (using a path you know doesn't exist).
  - **Expected**: `[warning]` on stderr; if no term given, no results returned (silently empty).
  - **Verifies**: warning emission for missing path.
- [ ] **3.5** `csb scan -d /nonexistent-dir amdead`.
  - **Expected**: `[warning]` on stderr, then results from broad-term search for "amdead".
  - **Verifies**: graceful fallback to term-mode when -d resolves to nothing.

## Section 4 -- `-D` (path-strict, exact folder only)

- [ ] **4.1** `csb scan -D amdead` from a directory containing `amdead/` with subdirs.
  - **Expected**: sessions whose start_folder OR top-N folder_usage paths EQUAL `<cwd>\amdead`. Sessions whose folder_usage points only at `<cwd>\amdead\sub` should NOT match (descendants excluded).
- [ ] **4.2** `csb scan -D amdead*` (wildcard with no descendants).
  - **Expected**: sessions matching `amdead`, `amdead-fork`, etc. -- but NOT their descendants like `amdead\sub` or `amdead-fork\sub`.
  - **Verifies**: NOT LIKE clause excludes descendants while LIKE allows siblings.

## Section 4b -- `-s` (start_folder only; "what originated here?")

- [ ] **4b.1** `csb scan -s C:\code` -- a directory many sessions start in.
  - **Expected**: only sessions whose `start_folder` is `C:\code` or a descendant. Sessions where `C:\code` appears only in `folder_usage` (e.g., a session that started elsewhere but did work under `C:\code`) should NOT match.
  - **Verifies**: `-s` skips folder_usage entirely.
- [ ] **4b.2** Compare: `csb scan -d C:\code -n 50` and `csb scan -s C:\code -n 50`.
  - **Expected**: `-d` returns a superset of `-s` (it matches via start_folder OR folder_usage; `-s` matches start_folder only).
- [ ] **4b.3** `csb scan -s amdead*` (wildcard).
  - **Expected**: sessions whose start_folder begins with `<cwd>\amdead` (sibling-prefix expansion via trailing `*`).
- [ ] **4b.4** `csb scan -s . ` (dot, expands to cwd).
  - **Expected**: sessions whose start_folder is cwd or a descendant. Useful as "is there anything I can resume that started in this folder?"

## Section 5 -- Combined: `-d`/`-D`/`-s` + term

- [ ] **5.1** `csb scan -d amdead my-paper` (path scope + term filter).
  - **Expected**: sessions in `<cwd>\amdead` (or descendants) that ALSO have "my-paper" somewhere in their metadata.
  - **Verifies**: post-filter by term within path-scoped results.
- [ ] **5.2** `csb scan -D amdead my-paper`.
  - **Expected**: same as 5.1 but only sessions whose path is exactly `<cwd>\amdead` (no descendants).
- [ ] **5.3** `csb scan -s amdead my-paper`.
  - **Expected**: sessions whose start_folder is `<cwd>\amdead` or descendants AND have "my-paper" in metadata.

## Section 6 -- Mutual exclusion + invalid input

- [ ] **6.1** `csb scan -d amdead -D redditslack`.
  - **Expected**: argparse error (mutex group). Exit code 2.
- [ ] **6.2** `csb scan -d` (no value for -d).
  - **Expected**: argparse error: "expected one argument".
- [ ] **6.3** `csb scan -D` (no value for -D).
  - **Expected**: argparse error: "expected one argument".
- [ ] **6.4** `csb scan -d amdead -s redditslack`.
  - **Expected**: argparse error (mutex group). Exit code 2.
- [ ] **6.5** `csb scan -D amdead -s redditslack`.
  - **Expected**: argparse error (mutex group). Exit code 2.
- [ ] **6.6** `csb scan -s` (no value for -s).
  - **Expected**: argparse error: "expected one argument".

## Section 7 -- Top-N gating

- [ ] **7.1** Identify a session whose folder_usage has the term you'll search at rank > 3 (e.g., the 5th-most-used folder for some session is `C:\code\amdead`).
  - Find with: `csb show <session-id>` and inspect the folder list.
- [ ] **7.2** `csb scan amdead`.
  - **Expected**: that session does NOT appear in results (rank 5 is below the default top-3 gate).
- [ ] **7.3** `csb scan amdead --all-folders`.
  - **Expected**: that session DOES appear (top-N gate removed).
- [ ] **7.4** `csb scan amdead --top 10`.
  - **Expected**: that session DOES appear (gate raised to top 10).

## Section 8 -- Path patterns with special characters

- [ ] **8.1** `csb scan -d "my_folder"` (path with literal underscore).
  - **Expected**: matches paths exactly `<cwd>\my_folder` -- NOT paths like `<cwd>\myXfolder` (underscore is escaped, not a SQL wildcard).
- [ ] **8.2** `csb scan -d "50%-share"` (path with literal percent).
  - **Expected**: matches paths exactly `<cwd>\50%-share` -- the `%` is treated as literal, not a SQL wildcard.
  - **Verifies**: `escape_like_value` correctly escapes `_` and `%`.

## Section 9 -- Compose with existing scan flags

- [ ] **9.1** `csb scan -d amdead --top 5`.
  - **Expected**: scan output shows top 5 folders per session row. The top-5 gate also applies to folder_usage matching for -d.
- [ ] **9.2** `csb scan -d amdead --all-folders`.
  - **Expected**: every folder shown per session row; no top-N gate on matching either.
- [ ] **9.3** `csb scan -d amdead -NU`.
  - **Expected**: only matches via start_folder; folder_usage matching skipped.
- [ ] **9.4** Cross-shell:
  - PowerShell: `csb scan -d amdead`
  - cmd.exe: `csb scan -d amdead`
  - WSL bash: `csb scan -d amdead` (with appropriate path translation)
  - **Expected**: same results, no shell-specific escaping issues.

## Section 10 -- Output format

- [ ] **10.1** Visual scan: confirm "Scanning for sessions ..." line at top of output reflects the chosen mode.
  - Bare: `Scanning for sessions under <cwd>...`
  - Term: `Scanning for sessions matching '<term>'...`
  - -d/-D/-s: `Scanning for sessions under <path>...`
  - -d/-D/-s + term: `Scanning for sessions under <path> filtered by '<term>'...`
  - -d/-D/-s wildcard: `Scanning for sessions matching pattern <path>*...`
- [ ] **10.2** "Found N session(s) ..." footer matches the same scope label.
- [ ] **10.3** Confirm result rows look identical to `csb list` output (same renderer, same `--top` honored).

## What mocks don't cover

This checklist exists because automated tests can't catch:

1. **Terminal rendering quirks**: Rich-vs-plain output, color, alignment, line wrapping in narrow terminals.
2. **Stderr/stdout split**: hint and warning lines must NOT contaminate stdout when piping or redirecting.
3. **Real subprocess invocation**: parser tests use `parser.parse_args()` directly, not via subprocess. End-to-end CLI invocation could fail on Python entry-point resolution.
4. **Cross-shell behavior**: PowerShell, cmd.exe, WSL bash all handle quoting differently. The same command from different shells should produce identical results.
5. **Real SQLite + real index**: integration tests use mock_db fixtures; this checklist exercises the actual `~/.claude/session-backup.db` against real session data.
6. **Discoverability**: does `csb scan --help` clearly explain when to use `-d` vs `-D` vs term? Read it from a fresh-eyes perspective.
7. **Refactor regression**: `cmd_scan` was substantially rewritten. Confirm `csb backup` and `csb list` still work normally (sanity).
