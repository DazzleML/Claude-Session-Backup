# Human Test Checklist: transcript_walker + strength column (v0.3.1)

**Target version:** v0.3.1
**Ships with commit:** v0.3.1: refactor: shared transcript_walker + file-op strength + JSONL parity
**Companion automated tests:** 9 new (5 in `test_fts5_db.py`, 4 in `test_search.py`). Total 496/496 pass.

## What this is

Hand-runnable verification for v0.3.1 -- the parity-foundation step of issue #3's v0.3.x track. Three things to confirm:

1. **JSONL search now surfaces sub-agent content** -- pre-v0.3.1, `csb search --source jsonl` silently missed Task-launched Agent output (Explore, oracle, Plan, etc.) and skill-attributed assistant events.
2. **Existing per-project FTS5 DBs migrate in place** -- v0.3.0 DBs (no `strength` column, no version table) get an `ALTER TABLE` + backfill on first open under v0.3.1. Nothing rebuilds; nothing breaks.
3. **`csb build-fts5` produces strength values** for new imports.

`csb search` still does NOT query FTS5 (that's v0.3.2). v0.3.1 ships the data layer only.

## High-Value Verification (~5 minutes)

### HV.1 -- Existing v0.3.0 DBs migrate to v2 on open

- [ ] **Steps (PowerShell):**
  ```powershell
  python -c "
  import sqlite3, glob
  for db in glob.glob(r'$env:USERPROFILE/.claude/csb-fts/*.db'):
      c = sqlite3.connect(db)
      cols = [r[1] for r in c.execute('PRAGMA table_info(file_operations)').fetchall()]
      has_strength = 'strength' in cols
      print(f'{db.split(\"\\\\\")[-1][:50]}  strength={has_strength}')
      c.close()
  "
  ```
- [ ] **Expected (before any v0.3.1 csb command):** Most rows say `strength=False` (these are v0.3.0 DBs). At least one is `False` so the migration can be tested.
- [ ] **Trigger migration:** `csb build-fts5 --quiet` (or `csb build-fts5 --project <slug>` to scope to one DB).
- [ ] **Re-run the inspection.** All `strength=True`. No errors.

### HV.2 -- strength values are populated correctly

- [ ] **Steps:**
  ```powershell
  $db = (Get-ChildItem ~/.claude/csb-fts/*.db | Select-Object -First 1).FullName
  python -c "
  import sqlite3
  c = sqlite3.connect(r'$db')
  for op, strength, cnt in c.execute('SELECT operation, strength, COUNT(*) FROM file_operations GROUP BY operation, strength ORDER BY strength DESC, operation').fetchall():
      print(f'  {op:>16}  strength={strength}  count={cnt}')
  "
  ```
- [ ] **Expected:**
  - `wrote`, `edited`, `notebook_edit` rows have `strength=3`
  - `read` rows have `strength=2`
  - `searched` rows have `strength=1`

### HV.3 -- `fts_schema_version` table stamped at 2

- [ ] **Steps:**
  ```powershell
  python -c "
  import sqlite3
  c = sqlite3.connect(r'$db')
  v = c.execute('SELECT version FROM fts_schema_version').fetchone()[0]
  print(f'fts_schema_version = {v}  (expected 2)')
  "
  ```
- [ ] **Expected:** `fts_schema_version = 2`

### HV.4 -- `--source jsonl` surfaces skill-attributed AGENT content

- [ ] **Steps (replace session-id with one of YOUR session IDs from `csb list -n 5`):**
  ```
  csb search "fullpostmortem" --source jsonl --session-id <prefix> --limit 3
  ```
- [ ] **Expected:** at least one hit with role tag `[AGENT:fullpostmortem]` (or another skill name). Pre-v0.3.1, those rows showed up as `[AI]` and the attribution was lost.

### HV.5 -- `--source jsonl` surfaces Task-launched Agent content

- [ ] **Steps:** pick a search term you know came from an Explore agent's report (e.g. "Bottom-line conclusion", "Surgery Surface Map", or any phrase you remember an agent wrote). Then:
  ```
  csb search "<phrase>" --source jsonl --limit 3
  ```
- [ ] **Expected:** hit with role tag `[AGENT:explore]` (or `[AGENT:oracle]`, etc.). Pre-v0.3.1 this would have returned nothing -- the Phase 1 parser ignored `tool_result` blocks.

### HV.6 -- `csb build-fts5` writes strength on new imports

- [ ] **Steps:** force-reindex a single session and confirm strength is written.
  ```powershell
  $sid = (csb list -n 1 --json | ConvertFrom-Json)[0].session_id.Substring(0, 8)
  csb build-fts5 --session-id $sid --force
  python -c "
  import sqlite3
  c = sqlite3.connect(r'$db')
  rows = c.execute(\"SELECT operation, strength FROM file_operations WHERE session_id LIKE '$sid%' LIMIT 5\").fetchall()
  print(rows)
  "
  ```
- [ ] **Expected:** every row has a non-null `strength` value in {1, 2, 3}.

## Reset between sections

If you want to test the v0.3.0→v0.3.1 migration twice, restore the original v0.3.0 DBs:
```powershell
Remove-Item ~/.claude/csb-fts/*.db
csb build-fts5  # rebuilds fresh v2 DBs (not actually v0.3.0 layout anymore)
```
True v0.3.0 layout reproduction requires checking out commit `ff8c497`, running `csb build-fts5`, then checking out v0.3.1 -- one-time-only test.
