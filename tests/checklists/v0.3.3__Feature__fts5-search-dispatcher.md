## What this is

The keystone of the v0.3.x track: `csb search` now actually queries FTS5. This checklist verifies the user-visible behavior end-to-end on real data.

## Prerequisites

1. csb v0.3.3 installed (`pip install -e ".[dev]"`)
2. `csb build-fts5` has been run at least once recently (within minutes of the test, so JSONLs haven't drifted)
3. At least one session in the local vault used a Task-launched sub-agent (Explore / oracle / Plan) so the AGENT role label can be verified end-to-end

## High-Value Verification (~5 minutes)

### HV.1 -- `--source fts5` returns an indexed AGENT row

- [ ] **Steps:** find a phrase you know came from an Explore agent or other sub-agent in your indexed sessions, then:
  ```
  csb search "<phrase>" --source fts5 --limit 2
  ```
- [ ] **Expected:** at least one hit with role tag `[AGENT:explore]` (or other subtype) returned via FTS5.

### HV.2 -- `--source fts5` skips unindexed sessions

- [ ] **Steps:** find a phrase that exists in a session you have NEVER run `csb build-fts5` against (or simulate by removing one project DB):
  ```powershell
  $db = (Get-ChildItem ~/.claude/csb-fts/<some-project>__*.db).FullName
  Move-Item $db "$db.bak"  # temporarily hide
  csb search "<phrase that was only in that project>" --source fts5
  Move-Item "$db.bak" $db  # restore
  ```
- [ ] **Expected:** "No content matches" (clean failure mode -- no grep fallback under explicit `fts5`).

### HV.3 -- Default `auto` mode uses FTS5 when fresh

- [ ] **Steps:**
  ```
  csb build-fts5 --quiet
  csb search "<phrase you know is in an indexed session>" --limit 1 --json | python -c "import json, sys; print(json.load(sys.stdin)[0]['source_type'])"
  ```
- [ ] **Expected:** `fts5` (the auto dispatcher picked the index over grep).

### HV.4 -- Default `auto` mode falls through to grep when stale

- [ ] **Steps:** modify a JSONL after the last build so its mtime advances past the indexed mtime:
  ```powershell
  $jsonl = (Get-ChildItem ~/.claude/projects/C--code-claude-projects-Claude-Session-Backup/*.jsonl | Select-Object -First 1).FullName
  Add-Content -Path $jsonl -Value ''   # tickle mtime
  csb search "<phrase that's in the existing convo log>" --source convo --limit 1 --json | python -c "import json, sys; print(json.load(sys.stdin)[0]['source_type'])"
  # Now run with default auto:
  csb search "<same phrase>" --limit 1 --json | python -c "import json, sys; print(json.load(sys.stdin)[0]['source_type'])"
  ```
- [ ] **Expected:** the second command's source_type is NOT `fts5` (e.g. `convo` / `sesslog` / `jsonl`). Auto detected the stale index and fell through.

### HV.5 -- Performance: FTS5 is fast

- [ ] **Steps:**
  ```powershell
  csb build-fts5 --quiet
  Measure-Command { csb search "function" --source fts5 --limit 50 | Out-Null } | Select-Object TotalMilliseconds
  ```
- [ ] **Expected:** under ~500ms on a vault with 100+ sessions. Grep equivalent would be in the seconds range.

### HV.6 -- Help text mentions the new choice

- [ ] **Steps:**
  ```
  csb search --help
  ```
- [ ] **Expected:** `--source` choices include `fts5` and the help text mentions the smart-fallback behavior.

## Section 1: Source-override semantics

### 1.1 `--source convo` unchanged

- [ ] `csb search "PUVM" --source convo --limit 2` -- output identical to v0.3.2; never touches FTS5.

### 1.2 `--source sesslog` unchanged

- [ ] Same as above with `--source sesslog`.

### 1.3 `--source jsonl` unchanged (still walks JSONL with v0.3.1 parity)

- [ ] `csb search "Surgery Surface" --source jsonl --limit 2` -- AGENT:explore content surfaces (v0.3.1 fix).

### 1.4 `--source fts5` does NOT trigger a migration

- [ ] If you have a v0.3.0-era DB on disk that hasn't been migrated yet (unlikely now -- v0.3.1 migrated everything on first open), a `csb search --source fts5` against it should NOT print any `csb: per-project FTS5 schema migrated` notice. The search opens FTS5 DBs raw to avoid touching schema.

## Section 2: AGENT role label preservation through FTS5

### 2.1 FTS5 returns `AGENT:<subtype>` not bare AGENT

- [ ] `csb search "Surgery Surface Map" --source fts5 --limit 1` -- the role tag should be `[AGENT:explore]` (or whatever subtype actually owns the content), not `[AGENT]` or `[AI]`.

### 2.2 AGENT subtype is searchable as part of the matched line

- [ ] `csb search "AGENT:explore" --source fts5 --limit 2` -- since the role label appears in the rendered output, FTS5 + Python matcher should surface lines whose role tag matches. Useful proxy until a proper `--role` flag exists.

---

## Section 3: Mixed-mode sanity

### 3.1 Two sessions, one indexed (FTS5), one not (grep)

- [ ] Find a phrase that appears in two sessions where one is indexed and one isn't:
  ```
  csb search "<phrase>" --limit 5
  ```
- [ ] **Expected:** both sessions return hits; the JSON `source_type` differs (`fts5` for one, `convo`/`jsonl` for the other).

### 3.2 `--limit` works correctly across mixed sources

- [ ] `csb search "function" --limit 3` -- exactly 3 hits returned, regardless of which source each came from.

---

## Section 4: What v0.3.3 deliberately does NOT do

- [ ] `csb files <pattern>` -- still not a command (v0.3.x deferred).
- [ ] `csb search -d/-D <path>` -- still not a flag (v0.3.x deferred).
- [ ] FTS5 ranking by `bm25()` -- not used; default insertion-order ranking via `ORDER BY message_index` matches Phase 1 semantics.
- [ ] Cross-project FTS5 query in a single SQL -- not done; we still iterate per project DB.
