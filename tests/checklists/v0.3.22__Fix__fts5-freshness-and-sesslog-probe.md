# Human Test Checklist: FTS5 Content-Hash Freshness + Sesslog Conversation Probe

**Target version:** v0.3.22
**Ships with commit:** `v0.3.22: fix: csb search -- content-hash freshness rescue + sesslog conversation probe (Closes #36)`
**Companion automated tests:** 4 new in `tests/test_search.py` (#36 section); 884 total in the suite.

## What this is

Two search-dispatch robustness fixes (#36): auto-search no longer abandons a current FTS5 index just because a byte-identical rewrite moved the file's mtime, and a shell-only `.sesslog` no longer dead-ends the dispatch before the `jsonl` that has the content. This checklist covers the real-data behaviors: search latency feel on the hash-rescue path and the incident-replay scenario.

## Prerequisites

1. csb v0.3.22 (`csb --version` shows `ALPHA 0.3.22`).
2. A session with FTS5 built (`csb update build-fts5 --session-id <prefix>` if needed) and a known phrase in it.
3. Steps in Section 1 temporarily touch a transcript's mtime -- they restore it afterward; have the session's prefix handy.

## High-Value Verification (~3 minutes)

### HV.1 -- Incident replay: byte-identical mtime bump does not break search

Validates: the exact b6a4929f false-negative (executed live during development; re-runnable).

- [ ] **Steps** (cross-shell, Python for the reversible mtime bump):
  ```
  python -c "import os,time; p=r'<abs-path-to-a-fts5-indexed-jsonl>'; st=os.stat(p); os.utime(p,(time.time(),time.time())); print('bumped')"
  csb search "<known-phrase>" --session <prefix>
  python -c "import os; p=r'<same-path>'; os.utime(p,(<orig-atime>,<orig-mtime>)); print('restored')"
  ```
  (Capture `os.stat(p).st_mtime` BEFORE bumping so you can restore it.)
- [ ] **Expected:** the search finds the phrase (source fts5) -- NOT "No content matches". Latency: barely slower than a normal search (one file hash on the stale path).
- [ ] **Why this matters:** this was a real user-facing false-negative after every restore/rsync until v0.3.18+v0.3.22.

### HV.2 -- Real content change still re-greps (no false-fresh)

- [ ] **Steps:** append a line containing a NEW unique phrase to a copy-scratch session's JSONL (use a scratch claude-dir, not your real one), then `csb search <new-phrase> --claude-dir <scratch> --db <scratch-db>`.
- [ ] **Expected:** the new phrase IS found (hash mismatch -> falls through to live grep). The rescue never serves stale content.

### HV.3 -- Shell-only sesslog session finds its jsonl content

- [ ] **Steps:** identify a session whose only logger channel is `.sesslog_bash` (no `.convo*`): `csb search "<phrase-known-in-its-transcript>" --session <prefix>`
- [ ] **Expected:** matches from the `jsonl` source -- the dispatch walked past the block-less sesslog instead of returning nothing.

### HV.4 -- Explicit --source contracts unchanged

- [ ] **Steps:** `csb search "<phrase>" --session <prefix> --source sesslog` on that same shell-only session; and `--source fts5` on a stale-indexed session.
- [ ] **Expected:** sesslog-explicit returns no matches (no fall-through); fts5-explicit returns whatever the index has (freshness skipped) -- both exactly as documented.
