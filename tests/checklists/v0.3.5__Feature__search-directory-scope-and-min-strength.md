# Human Test Checklist: csb search -d/-D directory-scope + --min-strength (v0.3.5)

**Target version:** v0.3.5
**Ships with commit:** v0.3.5: feat: csb search -d/-D dir-scope + --min-strength
**Companion automated tests:** 15 new in `tests/test_search.py`. Total 562/562 pass.

## What this is

Three things to confirm by hand:

1. **`-d <path>` recursive directory-scope ranks sessions by file-op strength** -- the session that most heavily worked on files under PATH appears first; the strength signal is visible in the session header.
2. **`-D <path>` folder-only excludes subdirectory matches** -- sessions whose only hits are in subfolders of PATH drop out.
3. **`--min-strength {1,2,3}` filters file-op rows by importance tier** -- `2` drops Grep/Glob probes, `3` keeps only active-modification rows (edited / wrote / notebook_edit).

Argparse mutex (`-d` xor `-D`) and runtime mutex (`-d/-D` rejects `--source jsonl|convo|sesslog`) are also covered here. Output across `--json` / `--sessions-only` / human modes is consistent in carrying the per-session strength suffix.

## High-Value Verification (~5 minutes)

### HV.1 -- `-d` returns ranked sessions with strength suffix

- [ ] **Steps (PowerShell)**:
  ```powershell
  csb search "FTS5" -d "C:\code\claude-projects\Claude-Session-Backup\claude_session_backup" --limit 3
  ```
- [ ] **Expected:** at least one session block. The session header line should END with `  [N file-ops, strength=S]` where N and S are positive integers. Header looks like:
  ```
  <name>  <UUID>  (<project>)  today (...)  [149 file-ops, strength=377]
  ```
- [ ] **Why it matters:** confirms the dir-scope dispatcher ran (vs falling through to normal search) and the per-session strength carried into `Hit.strength_sum` / `Hit.file_op_count` for the renderer.

### HV.2 -- `--min-strength 3` reduces the file-op count and strength

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d "C:\code\claude-projects\Claude-Session-Backup\claude_session_backup" --sessions-only --limit 1
  csb search "FTS5" -d "C:\code\claude-projects\Claude-Session-Backup\claude_session_backup" --min-strength 3 --sessions-only --limit 1
  ```
- [ ] **Expected:** the second run shows the SAME top session but with a smaller `[N file-ops, strength=S]` (the strength-1 Grep/Glob rows and strength-2 read rows have been excluded from the SUM).
- [ ] **Why it matters:** confirms the WHERE clause `AND strength >= ?` is taking effect at step 1 of the dispatch.

### HV.3 -- `-d` and `-D` are mutually exclusive (argparse)

- [ ] **Steps:**
  ```powershell
  csb search foo -d . -D .
  ```
- [ ] **Expected:** argparse error and non-zero exit:
  ```
  csb search: error: argument -D/--directory-only: not allowed with argument -d/--directories-below
  ```
- [ ] **Why it matters:** the mutex group prevents users from specifying contradictory semantics.

### HV.4 -- `-d` rejects `--source jsonl` with a clear error

- [ ] **Steps:**
  ```powershell
  csb search foo -d . --source jsonl
  ```
- [ ] **Expected:** clear runtime error on stderr (not a Python traceback) and exit code 2:
  ```
  Error: -d/-D directory-scope is incompatible with --source jsonl. -d/-D requires FTS5; omit --source (auto resolves to fts5) or pass --source fts5.
  ```
- [ ] **Why it matters:** dir-scope only queries the per-project FTS5 DBs. We surface that constraint instead of silently producing zero hits.

### HV.5 -- `--json` carries `strength_sum` and `file_op_count`

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d "C:\code\claude-projects\Claude-Session-Backup\claude_session_backup" --json --limit 1 | python -m json.tool
  ```
- [ ] **Expected:** the printed JSON object includes `"strength_sum": N` and `"file_op_count": M` keys alongside the usual `session_id` / `matched_text` / etc.
- [ ] **Why it matters:** programmatic consumers can read the ranking signal without parsing the human-mode suffix.

### HV.6 -- Normal search (no `-d`) doesn't print the strength suffix

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" --limit 1
  ```
- [ ] **Expected:** the session header line ends with the relative-date tag (`today (...)`) and NO `[N file-ops, strength=S]` suffix.
- [ ] **Why it matters:** the new fields default to 0 on non-dir-scope hits; the renderer must suppress the suffix in that case.

## Section 1: directory-scope semantics

### 1.1 `-d` is recursive (folder + descendants)

- [ ] Pick a folder that has both immediate-child file-ops AND subdirectory file-ops across past sessions. Run:
  ```powershell
  csb search "" -d "<folder>" --sessions-only --limit 5
  ```
- [ ] **Expected:** sessions that touched ANY file under `<folder>` (including in subdirectories like `<folder>/tests/`) appear, ranked by file-op strength.

### 1.2 `-D` excludes subdirectory matches

- [ ] Same folder as 1.1, but with `-D`:
  ```powershell
  csb search "" -D "<folder>" --sessions-only --limit 5
  ```
- [ ] **Expected:** the count of sessions and/or the per-session strength values are LOWER than with `-d` -- any session whose hits were exclusively in subdirectories drops out entirely.

### 1.3 Relative paths resolve via `Path.resolve()`

- [ ] From a known cwd, run with a relative path:
  ```powershell
  cd C:\code\claude-projects\Claude-Session-Backup
  csb search "FTS5" -d claude_session_backup --limit 2
  ```
- [ ] **Expected:** same behavior as passing the full absolute path. The resolved path appears (implicitly) in the matched GLOBs.

### 1.4 Pattern with trailing separator works

- [ ] Compare:
  ```powershell
  csb search "FTS5" -d "C:\code\claude-projects\Claude-Session-Backup\claude_session_backup\" --limit 1
  csb search "FTS5" -d "C:\code\claude-projects\Claude-Session-Backup\claude_session_backup"  --limit 1
  ```
- [ ] **Expected:** identical results. Trailing `\` or `/` is stripped before pattern construction.

## Section 2: --min-strength filter

### 2.1 `--min-strength 1` (default) is a no-op

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d <folder> --sessions-only --limit 3
  csb search "FTS5" -d <folder> --min-strength 1 --sessions-only --limit 3
  ```
- [ ] **Expected:** identical output (every file-op row has strength >= 1, so the filter is inert).

### 2.2 `--min-strength 2` drops Grep/Glob probes

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d <folder> --min-strength 2 --sessions-only --limit 3
  ```
- [ ] **Expected:** strength counts decrease vs. `--min-strength 1`. The decrease equals the number of Grep/Glob rows for that session (strength=1 each).

### 2.3 `--min-strength 3` keeps only active-modification ops

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d <folder> --min-strength 3 --sessions-only --limit 3
  ```
- [ ] **Expected:** even more reduction. Sessions whose ONLY work was reading or searching files in `<folder>` (no Edit/Write/NotebookEdit) drop out entirely. Surviving sessions show strength values that are multiples of 3 (the strength weight of active mods).

### 2.4 `--min-strength` outside `-d/-D` is silently ignored

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" --min-strength 3 --limit 1
  ```
- [ ] **Expected:** normal results, no error, no warning. The flag only takes effect in dir-scope mode.

## Section 3: cross-mode integration

### 3.1 `--sessions-only` in dir-scope mode

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d <folder> --sessions-only
  ```
- [ ] **Expected:** one-line-per-session summary, each with the `[N file-ops, strength=S]` suffix appended after the hit count.

### 3.2 `--shortid` works in dir-scope mode

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d <folder> --shortid --limit 2
  ```
- [ ] **Expected:** compact `<head>-...-<tail>` UUID display in headers. Strength suffix still present.

### 3.3 `--session-id <prefix>` filter combines with `-d`

- [ ] Pick a UUID prefix from a session known to have hits under `<folder>`:
  ```powershell
  csb search "FTS5" -d <folder> --session-id <8-char-prefix> --limit 5
  ```
- [ ] **Expected:** results constrained to that one session (or empty if the prefix matches a session that didn't touch `<folder>`).

### 3.4 `-A 2 -B 1` context windows work in dir-scope mode

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d <folder> -A 2 -B 1 --limit 1
  ```
- [ ] **Expected:** matched line PLUS one event above and two below, same as the non-dir-scope context behavior.

### 3.5 `--limit N` caps hits in dir-scope mode

- [ ] **Steps:**
  ```powershell
  csb search "" -d <folder> --limit 2
  ```
- [ ] **Expected:** at most 2 hits total (NOT 2 per session). The limit fires at the global level after strength ordering.

## Section 4: tolerance / edge cases

### 4.1 Empty pattern matches every event in ranked sessions

- [ ] **Steps:**
  ```powershell
  csb search "" -d <folder> --sessions-only --limit 3
  ```
- [ ] **Expected:** the top 3 sessions (by strength) appear, with hit counts equal to the total events in each session (or capped by `--limit`).

### 4.2 No csb-fts directory -> empty result

- [ ] If you can set up a fresh `--claude-dir` pointing to a `~/.claude` clone WITHOUT a `csb-fts/` subdirectory, run a dir-scope query. **Expected:** no crash, no hits, exit code 0.

### 4.3 Path that no session ever touched

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -d C:\path\that\does\not\exist --limit 5
  ```
- [ ] **Expected:** "No content matches for 'FTS5'" and the usual hint line. No crash.

### 4.4 Pattern with regex chars + literal mode

- [ ] **Steps:**
  ```powershell
  csb search "foo.bar" -d <folder> --limit 2
  ```
- [ ] **Expected:** literal substring match (the `.` does NOT match arbitrary chars). Confirmed in matched-line previews.

### 4.5 Pattern with regex chars + `-E` regex mode

- [ ] **Steps:**
  ```powershell
  csb search "foo.bar" -d <folder> -E --limit 2
  ```
- [ ] **Expected:** regex semantics -- the `.` matches any single character. Confirmed in matched-line previews.

## Section 5: query-match highlighting (human mode)

### 5.1 Matched query renders in bold green (visual check)

- [ ] **Steps (real terminal -- NOT piped, color must be enabled):**
  ```powershell
  csb search "FTS5" --limit 1
  ```
- [ ] **Expected:** every occurrence of `FTS5` inside the `>` excerpt line and any context lines is **bold green** while the rest of the text is its default terminal color.
- [ ] **Why it matters:** the whole point of the feature is at-a-glance visibility of why the line matched.

### 5.2 Highlight respects case-sensitivity

- [ ] **Steps:**
  ```powershell
  csb search "fts5" --limit 1                  # default: case-insensitive
  csb search "fts5" --limit 1 -s               # case-sensitive
  ```
- [ ] **Expected:** the first command highlights both `FTS5` and `fts5` occurrences (lowercase pattern matches uppercase); the second highlights only literal `fts5` (mixed-case occurrences in the excerpt go unhighlighted).

### 5.3 Highlight works on regex matches

- [ ] **Steps:**
  ```powershell
  csb search "FTS|fts" -E --limit 1
  ```
- [ ] **Expected:** both `FTS` and `fts` substrings highlighted independently inside any matched line.

### 5.4 `--no-color` disables highlight

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" --no-color --limit 1
  ```
- [ ] **Expected:** plain text only. No ANSI codes anywhere in the output (verify by piping through `Out-File temp.txt; type temp.txt` or similar).

### 5.5 Highlight survives `-A`/`-B` context

- [ ] **Steps:**
  ```powershell
  csb search "FTS5" -A 2 -B 1 --limit 1
  ```
- [ ] **Expected:** the matched line AND any context lines that incidentally contain `FTS5` both get the highlight.

## Section 6: --deleted {only, all} (v0.3.5 breaking change)

### 6.1 Bare `--deleted` defaults to `only`

- [ ] **Steps:**
  ```powershell
  csb search "" --deleted --sessions-only --limit 5
  ```
- [ ] **Expected:** only sessions marked deleted appear (or no hits if you have none deleted). NO live sessions in the result.

### 6.2 `--deleted only` is the explicit form

- [ ] **Steps:**
  ```powershell
  csb search "" --deleted only --sessions-only --limit 5
  ```
- [ ] **Expected:** identical to 6.1.

### 6.3 `--deleted all` shows live + deleted

- [ ] **Steps:**
  ```powershell
  csb search "" --sessions-only --limit 5                # live only
  csb search "" --deleted all --sessions-only --limit 5  # live + deleted
  ```
- [ ] **Expected:** the second command shows >= the first (any deleted sessions added in). If you have no deleted sessions, the two outputs match.

### 6.4 Invalid subvalue rejected with clear error

- [ ] **Steps:**
  ```powershell
  csb search foo --deleted maybe
  ```
- [ ] **Expected:**
  ```
  csb search: error: argument --deleted: invalid choice: 'maybe' (choose from 'only', 'all')
  ```

### 6.5 `csb list` mirrors the same shape

- [ ] **Steps:**
  ```powershell
  csb list -n 5                       # default: live only
  csb list --deleted -n 5             # only deleted
  csb list --deleted only -n 5        # only deleted (explicit)
  csb list --deleted all -n 5         # live + deleted
  ```
- [ ] **Expected:** same semantics as `csb search`. Argparse error on `--deleted maybe`.

### 6.6 Old `--all` flag is gone (breaking)

- [ ] **Steps:**
  ```powershell
  csb search foo --all
  csb list --all
  ```
- [ ] **Expected:** both report `unrecognized arguments: --all` and exit 2. (Shell scripts using `--all` must migrate to `--deleted all`.)
