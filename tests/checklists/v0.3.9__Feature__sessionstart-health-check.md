# Human Test Checklist: SessionStart health check (v0.3.9)

**Target version:** v0.3.9
**Ships with commit:** v0.3.9: feat: SessionStart health check (detect un-backed-up sessions)
**Companion automated tests:** detector (4 in `test_commands.py`) + SessionStart-detector flow (4 in `test_backup_hook.py`). Total 631/631 pass.

## What this is

SessionStart no longer backs up unconditionally. It now runs an internal detector (`csb _check`, hidden from `csb --help`) to find a *prior* session with un-backed-up changes (an unclean shutdown where SessionEnd never ran); only then does it surface a `systemMessage` warning **and** run a recovery backup. The clean path does nothing. `_check` is a hook mechanism, not a user-facing command -- but maintainers can run it by hand for post-crash triage. Mocks can't fire real hooks or show real `systemMessage` UI, so this verifies the end-to-end behavior by hand.

## High-Value Verification (~5 min)

### HV.1 -- `csb _check` reports clean on a healthy vault
- [ ] Run `csb backup` (bring the index current), then `csb _check`.
- [ ] **Expected:** `All sessions backed up.` and exit code 0 (`echo $LASTEXITCODE` in PowerShell / `echo $?` in bash).

### HV.2 -- `csb _check` detects a gap (exit 10)
- [ ] Make a session look un-backed-up: the simplest real trigger is to have an active session (its JSONL is newer than the last backup). Run `csb _check` **without** `--exclude`.
- [ ] **Expected:** `csb: N session(s) with un-backed-up changes ...` listing short ids, exit code **10**.
- [ ] Now `csb _check --exclude <that-session-uuid>` -> `All sessions backed up.`, exit 0. (This is exactly what the hook does for the current session.)

### HV.3 -- Normal SessionStart is silent (no backup, no warning)
- [ ] Exit a session cleanly (SessionEnd backs it up via v0.3.8), then start a new one.
- [ ] `Get-Content "$env:USERPROFILE\.claude\csb-logs\backup-SessionStart.log" -Tail 3`
- [ ] **Expected:** a line like `SessionStart source=startup: all sessions backed up -- no action`. **No** new backup spawned, **no** warning shown in the session.

### HV.4 -- SessionStart warns + recovers after an unclean shutdown
- [ ] Simulate a missed SessionEnd: in a session, do distinctive work, then kill the terminal window abruptly (or `kill -9` the process) so SessionEnd's backup is interrupted before it commits. Start a new session.
- [ ] **Expected:** Claude Code shows a `systemMessage` like *"csb: 1 session(s) with un-backed-up changes ..."*, and `backup-SessionStart.log` shows `GAP detected -> warn user + recover backup`. Shortly after, `csb _check` is clean (the recovery backup ran).
- [ ] **Why it matters:** this is the whole point -- a missed backup is surfaced, not silently masked, and still recovered.

### HV.5 -- `csb status` shows the `Un-backed-up:` line
- [ ] Run `csb status`.
- [ ] **Expected:** an `Un-backed-up:` line. If run inside an active session, it lists that session honestly (`<id>  <name>  (changed since last backup)`); from a plain terminal after `csb backup`, it reads `none`.
- [ ] Tweak the cap: set `"status_unbacked_limit": 0` in `~/.claude/session-backup-config.json`, re-run `csb status` -> the list collapses to `+ N more not shown`. Restore the file after.
- [ ] **Why it matters:** this is the user-facing "did my session work get saved?" answer (issue #5), distinct from the git-level `Uncommitted changes` count.

## Section 1: `csb _check` semantics (no session needed)

### 1.1 exit codes
- [ ] Clean vault: `csb _check; echo exit=$LASTEXITCODE` -> `exit=0`.
- [ ] With an un-backed-up session: `exit=10`.
- [ ] `csb _check --quiet` on a clean vault prints nothing (still exit 0); on a gap it still prints the summary (the summary is the point).
