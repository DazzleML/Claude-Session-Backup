# Human Test Checklist: `csb config settings:` TTL passthrough

**Target version:** 0.3.10
**Ships with commit:** TBD (feat: expose Claude Code purge TTL via `csb config settings:cleanupPeriodDays`)
**Companion automated tests:** 36 tests in `tests/test_config.py` (config.py helpers + `cmd_config` dispatch via `cli.main`). Full suite: 672 passing.

## What this is

`csb` can now show and change Claude Code's session purge TTL (`cleanupPeriodDays`) without you hand-editing `settings.json`. The TTL is addressed through a fully-qualified `settings:` namespace so it can never be confused with csb's own config: a **bare** key (`csb config display_top_folders`) always means csb's `session-backup-config.json`; a **`settings:`** key (`csb config settings:cleanupPeriodDays`) always means Claude Code's `settings.json`.

This checklist verifies the user-visible behavior a human must eyeball: the GET output and guidance, the SET write actually landing in the right file while preserving everything else, the `--force` guard on the destructive `0` value, the reduction warning, and the error/redirect messages. Mocks confirm the logic; only a human confirms the wording reads well and the real file on disk is intact.

## Prerequisites

1. **Environment:** Python 3.10+, this repo installed editable (`pip install -e ".[dev]"`) so `csb` and the package are importable. Windows, Linux, or macOS.
2. **State isolation — do NOT touch your real `~/.claude`.** Every command below uses `--claude-dir <throwaway>` pointing at a temp directory you create. The real `~/.claude/settings.json` is never read or written by this checklist.
3. **Reset between sections:** re-seed the throwaway `settings.json` (the "Setup" block below) to return to a known state.

### Setup — create a throwaway claude dir with a seeded settings.json

- [ ] **Steps** (pick your shell):

  **cmd.exe**:
  ```cmd
  set CSBTEST=%TEMP%\csb-ttl-check\.claude
  if exist "%CSBTEST%" rmdir /s /q "%CSBTEST%"
  mkdir "%CSBTEST%"
  > "%CSBTEST%\settings.json" echo {"cleanupPeriodDays": 90, "permissions": {"allow": ["Read", "Glob"]}, "model": "opus"}
  type "%CSBTEST%\settings.json"
  ```

  **PowerShell**:
  ```powershell
  $env:CSBTEST = "$env:TEMP\csb-ttl-check\.claude"
  if (Test-Path $env:CSBTEST) { Remove-Item $env:CSBTEST -Recurse -Force }
  New-Item -ItemType Directory -Force $env:CSBTEST | Out-Null
  '{ "cleanupPeriodDays": 90, "permissions": { "allow": ["Read", "Glob"] }, "model": "opus" }' | Out-File -Encoding utf8 "$env:CSBTEST\settings.json"
  Get-Content "$env:CSBTEST\settings.json"
  ```

  **POSIX**:
  ```bash
  export CSBTEST=/tmp/csb-ttl-check/.claude
  rm -rf "$CSBTEST"; mkdir -p "$CSBTEST"
  printf '{ "cleanupPeriodDays": 90, "permissions": { "allow": ["Read", "Glob"] }, "model": "opus" }\n' > "$CSBTEST/settings.json"
  cat "$CSBTEST/settings.json"
  ```

- [ ] **Expected:** a `settings.json` exists in `$CSBTEST` containing `cleanupPeriodDays: 90`, a `permissions` block, and `model: "opus"`.

> In the commands below, substitute your shell's variable for `$CSBTEST` (`%CSBTEST%` on cmd, `$env:CSBTEST` on PowerShell). The `csb` invocation itself is identical across shells.

## High-Value Verification (~5 minutes)

### HV.1 — GET shows current TTL + source + guidance

Validates: the read path resolves the right file and the human-facing explanation is correct and readable.

- [ ] **Steps:** `csb config settings:cleanupPeriodDays --claude-dir $CSBTEST`
- [ ] **Expected:** first stdout token is `90`, followed by the settings.json path. Guidance lines (default 30; how to change; the `0` caveat; the "never purge → large number" idiom) print below.
- [ ] **Why this matters:** this is the "make it easy to see how to change the TTL" goal — if the value or guidance is wrong, the feature fails its purpose.

### HV.2 — SET writes the value and preserves everything else

Validates: read-merge-write doesn't clobber the user's other settings.

- [ ] **Steps:**
  1. `csb config settings:cleanupPeriodDays 365 --claude-dir $CSBTEST`
  2. View `$CSBTEST/settings.json` (`type` / `Get-Content` / `cat`)
- [ ] **Expected:** `cleanupPeriodDays` is now `365`; `permissions` and `model` are **unchanged**; file is 2-space indented with a trailing newline.
- [ ] **Why this matters:** clobbering a user's `permissions`/`hooks` block would be a serious data-loss bug.

### HV.3 — The `0` guard refuses without `--force`

Validates: the destructive value is gated.

- [ ] **Steps:** `csb config settings:cleanupPeriodDays 0 --claude-dir $CSBTEST`
- [ ] **Expected:** non-zero exit (2), a message explaining `0` disables persistence / deletes transcripts, and pointing at the large-number idiom and `--force`. **The file is NOT changed** (still `365` from HV.2).
- [ ] **Why this matters:** `0` is a footgun (it means "delete everything", not "keep forever"); silently writing it would schedule deletion of all the user's transcripts.

### HV.4 — `--force` allows `0` with a loud warning

Validates: the escape hatch works and warns.

- [ ] **Steps:** `csb config settings:cleanupPeriodDays 0 --force --claude-dir $CSBTEST`
- [ ] **Expected:** exit 0, value written as `0`, and a WARNING that persistence is now off and csb backups remain recoverable.
- [ ] **Why this matters:** the user can still do the dangerous thing on purpose, but only with eyes open.

### HV.5 — Bare key redirects to the namespace

Validates: the disambiguation nudge.

- [ ] **Steps:** `csb config cleanupPeriodDays --claude-dir $CSBTEST`
- [ ] **Expected:** exit 1, a hint that `cleanupPeriodDays` is a Claude Code setting and to use `csb config settings:cleanupPeriodDays`.
- [ ] **Why this matters:** without the hint, a bare key would dead-end as "unknown config key" and confuse users.

### HV.6 — csb's own config is untouched by all of this

Validates: the two config files never cross-contaminate.

- [ ] **Steps:**
  1. `csb config display_top_folders 5 --claude-dir $CSBTEST`
  2. View `$CSBTEST/session-backup-config.json` and `$CSBTEST/settings.json`
- [ ] **Expected:** `display_top_folders: 5` lands in `session-backup-config.json`; it does **not** appear in `settings.json`. Conversely `cleanupPeriodDays` is not in `session-backup-config.json`.
- [ ] **Why this matters:** the whole design premise is that bare keys and `settings:` keys map to disjoint files.

After HV.1–HV.6 pass, the feature shipped correctly. The sections below cover error wording, the reduction warning, malformed-file safety, and the discovery surface.

## Section 1: GET behavior

### 1.1 GET when the key is absent → default

- [ ] **Steps:** remove the key (re-seed `settings.json` without `cleanupPeriodDays`, e.g. `{ "model": "opus" }`), then `csb config settings:cleanupPeriodDays --claude-dir $CSBTEST`
- [ ] **Expected:** stdout starts with `30` and the line notes `(default; not set in <path>)`.

### 1.2 GET stdout is parseable

- [ ] **Steps:** `csb config settings:cleanupPeriodDays --claude-dir $CSBTEST` and pipe stdout to a tool that reads the first token.

  **PowerShell**: `(csb config settings:cleanupPeriodDays --claude-dir $env:CSBTEST | Select-Object -First 1)`
  **POSIX**: `csb config settings:cleanupPeriodDays --claude-dir "$CSBTEST" | head -1 | awk '{print $1}'`
- [ ] **Expected:** the value is the leading token on stdout; guidance goes to **stderr** and does not pollute the piped value.

## Section 2: SET validation and warnings

### 2.1 Reduction warning

- [ ] **Steps:** with the value at `365`, run `csb config settings:cleanupPeriodDays 10 --claude-dir $CSBTEST`
- [ ] **Expected:** exit 0, value written `10`, plus a note that lowering the TTL may purge sessions older than 10 days at next Claude start, and that csb backups keep them recoverable.

### 2.2 Negative rejected

- [ ] **Steps:** `csb config settings:cleanupPeriodDays -5 --claude-dir $CSBTEST`
- [ ] **Expected:** exit 2, message that the value must be `>= 0` and to use a large number to never purge. File unchanged.

### 2.3 Non-integer rejected

- [ ] **Steps:** `csb config settings:cleanupPeriodDays abc --claude-dir $CSBTEST` and `... 1.5 ...`
- [ ] **Expected:** exit 2, "must be a whole number of days". File unchanged.

### 2.4 Never-purge idiom

- [ ] **Steps:** `csb config settings:cleanupPeriodDays 36500 --claude-dir $CSBTEST`
- [ ] **Expected:** exit 0, value `36500`. (Confirms there is no special "infinite" sentinel — a large number is the idiom.)

## Section 3: Safety / edge cases

### 3.1 Malformed settings.json is NOT clobbered

- [ ] **Steps:**
  1. Overwrite `$CSBTEST/settings.json` with invalid JSON (e.g. `{ broken`).
  2. `csb config settings:cleanupPeriodDays 365 --claude-dir $CSBTEST`
  3. View the file.
- [ ] **Expected:** exit 1, an error that the file is not valid JSON and csb refuses to overwrite it. **The broken content is left exactly as-is** (not replaced).

### 3.2 No settings.json at all → SET creates it

- [ ] **Steps:** delete `$CSBTEST/settings.json`, then `csb config settings:cleanupPeriodDays 200 --claude-dir $CSBTEST`, then view the file.
- [ ] **Expected:** a new `settings.json` is created containing exactly `{ "cleanupPeriodDays": 200 }`.

### 3.3 Unknown `settings:` key

- [ ] **Steps:** `csb config settings:bogusKey --claude-dir $CSBTEST`
- [ ] **Expected:** exit 1, message listing the keys csb can read/write and suggesting direct edit of settings.json for others.

## Section 4: Discovery

### 4.1 Bare `csb config` surfaces the namespace

- [ ] **Steps:** `csb config --claude-dir $CSBTEST`
- [ ] **Expected:** stdout is **pure JSON** (csb's own config — pipeable into `python -m json.tool`). stderr carries a "Claude Code settings (address with the settings: namespace)" footer listing `settings:cleanupPeriodDays`.

### 4.2 Help text documents the namespace

- [ ] **Steps:** `csb config -h`
- [ ] **Expected:** the description explains bare vs `settings:` keys and shows the TTL example; `--force` is listed with its danger note.

## Section 5: Cross-feature integration

### 5.1 The TTL drives the purge countdown

- [ ] **Steps:** set a known TTL (`csb config settings:cleanupPeriodDays 90 --claude-dir $CSBTEST`), then run `csb list --sort expiration --claude-dir $CSBTEST` against a dir that has sessions (or your real read-only listing in a separate, non-mutating check).
- [ ] **Expected:** the `(purge in Nd)` countdown reflects the TTL you set (same value `read_cleanup_period` consumes). Setting the TTL and the countdown stay consistent.

## What the automated tests DON'T cover

- **Output readability** — whether the GET guidance and SET warnings read clearly in a real terminal (wrapping, spacing).
- **Windows codepage rendering** — the `--` and quoting in messages on cmd.exe codepage 437/1252.
- **Real `settings.json` interaction** — the mocks use temp dirs; only a human confirms the round-trip against a real file with a large `permissions` block looks right and key order is preserved.
- **Discoverability** — can a user who knows "I want to change the purge time" find this via `csb config -h` / README?
- **Pipe behavior in a real shell** — that guidance truly stays on stderr and the value pipes cleanly (Section 1.2).
- **Claude Code actually honoring the written value** — verifying Claude Code reads the new TTL on next startup is out of scope for csb's tests.
