# Human Test Checklist: rebuild-index --include-fts5 Wiring

**Target version:** v0.4.4
**Ships with commit:** `v0.4.4: feat: wire rebuild-index --include-fts5 (Closes #3)`
**Companion automated tests:** real-build, fail-soft, and the two existing seam tests in `tests/test_restore.py`; 935 total.

## What this is

The last open AC of the Content Search epic: `csb update rebuild-index --include-fts5` was a documented no-op stub since v0.3.11. It now force wipes + rebuilds the per-project FTS5 DBs after the index rebuild, failing soft. This checklist covers the on-real-data feel and the one thing mocks can't show: wall-clock cost on a real vault.

## Prerequisites

## High-Value Verification (~5 min)

### HV.1 -- The wired flag on real data

- [ ] **Steps:** `csb --db %TEMP%\csb-044.db update rebuild-index --include-fts5`
- [ ] **Expected:** rebuild completes, then FTS5 progress output; `<claude-dir>/csb-fts/` afterwards contains per-project `.db` files with fresh mtimes. Note the wall-clock for the FTS5 stage on your vault size -- this is the opt-in cost of the flag.
- [ ] **Then:** `csb --db %TEMP%\csb-044.db search "<phrase you know exists>"` -- hits return (sub-second on indexed projects).

### HV.2 -- Without the flag, nothing FTS5 happens

- [ ] **Steps:** delete the scratch DB; `csb --db %TEMP%\csb-044.db update rebuild-index` (no flag); compare `csb-fts/*.db` mtimes
- [ ] **Expected:** FTS5 DB mtimes unchanged -- the refresh is strictly opt-in.

### HV.3 -- Fail-soft reads like a warning, not a failure

- [ ] **Eyeball check:** the warning text in `_maybe_refresh_fts5` (commands.py) names the manual fallback command and says the rebuild is intact. (Triggering it for real requires a no-FTS5 Python; the automated test monkeypatches it -- this item is a message-quality read.)

## What the automated tests DON'T cover

- Wall-clock cost of force-rebuild on a real-sized vault (HV.1)
- Search quality against the freshly force-rebuilt indexes (HV.1)
